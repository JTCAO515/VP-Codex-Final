# VisePanda Agent 交接包最后一层设计

## 概述

当前仓库已经有一套相对完整的交接包：

- `HANDOFF.md`：主交接入口
- `docs/2026-06-20-engineering-handoff-notes.md`：工程全景附录
- `docs/2026-06-20-first-week-takeover-checklist.md`：首周接手清单
- `docs/2026-06-20-high-risk-files-guide.md`：高风险文件修改指南
- `docs/2026-06-20-production-regression-manual.md`：线上回归检查手册
- `docs/2026-06-20-commercial-upgrade-plan.md`：商用升级路线

但如果接下来真正要交给**别的 coding agent** 持续接手，仅有“项目是什么”和“怎么回归”还不够。  
还需要补上最后一层更偏“判断与推进”的材料，让新的 agent 知道：

1. 接下来 2-4 周最合理的推进顺序是什么
2. 哪些技术债现在先不要碰
3. 从模块视角看，哪些边界最值得守住

这一层文档不能写成纯机械说明书，也不能写成纯背景故事，而应同时保留：

- 背景故事
- 当前判断
- 推荐动作
- 边界约束

## 目标

本轮完成后，仓库中的交接包需要进一步具备以下能力：

1. 新的 coding agent 不依赖聊天记录，也能理解“为什么项目会长成现在这样”
2. 新的 coding agent 能判断未来 2-4 周先做什么，而不是凭感觉开线
3. 新的 coding agent 知道哪些技术债应暂缓，哪些只适合顺手修
4. 新的 coding agent 知道按模块推进时，哪些边界不能轻易打破

## 非目标

本轮不做以下事情：

- 不修改产品功能
- 不修改测试代码
- 不改变部署形态
- 不调整版本号
- 不重写已有交接文档主体结构

本轮专注于**补齐更高层的 agent-oriented 交接材料**。

## 方案选择

### 方案 A：纯执行导向

文档只回答：

- 先做什么
- 后做什么
- 不要做什么

优点：

- 最容易直接执行

缺点：

- 新 agent 容易只照流程，不理解为什么要停下或换方向

### 方案 B：纯决策导向

文档只回答：

- 为什么当前判断是这样
- 哪些条件下才值得改优先级

优点：

- 更适合强 agent 自主判断

缺点：

- 起步不够直接

### 方案 C：混合版

本轮采用方案 C。

每份文档都同时保留：

1. 背景故事
2. 当前判断
3. 推荐动作
4. 边界说明

这样做的原因是，接手对象不是普通读者，而是新的 coding agent。  
它既需要“能立刻开始”，也需要“知道为什么现在不要乱改”。

## 设计方案

## 1. 未来 2-4 周推荐迭代顺序

新增文件：

`docs/2026-06-20-next-2-4-weeks-priority-guide.md`

### 目标

这份文档告诉接手 agent：

- 当前项目处在哪个阶段
- 接下来 2-4 周最合理的推进顺序是什么
- 为什么当前不建议优先大重构或整站迁移

### 结构

至少覆盖以下章节：

1. 文档目的
2. 背景
3. 当前判断
4. 推荐推进顺序
5. 为什么是这个顺序
6. 偏离顺序的触发条件

### 内容要求

必须保留项目背景故事，例如：

- 这个项目是从功能原型长出来的
- 后续逐步补了 Auth、Trips、Admin、英文原生化、Tools、移动端、稳定性
- 因此当前更重要的是继续收口与补全，而不是为了架构纯度推翻重来

## 2. 技术债边界说明

新增文件：

`docs/2026-06-20-technical-debt-boundaries.md`

### 目标

这份文档用于告诉接手 agent：

- 哪些技术债现在先接受
- 哪些债只能在特定条件成立时才值得动
- 哪些债可以在当前任务中顺手修

### 结构

至少覆盖以下章节：

1. 文档目的
2. 背景
3. 当前应接受的技术债
4. 当前不应主动触碰的技术债
5. 可以顺手修的小债
6. 什么时候技术债才值得升级为主任务

### 内容要求

必须明确包含以下问题的判断：

- `web/app.js` 过重，但当前不宜直接大拆
- `static/*` 兼容层存在，但当前不宜直接大清理
- `Vercel` 仍保留，不应因为局部问题立刻整站迁移
- auth / chat / trips 主链路不应在无明确目标下随意重构

## 3. 模块责任建议表

新增文件：

`docs/2026-06-20-module-ownership-guide.md`

### 目标

这份文档从“模块责任中心”的角度帮助接手 agent 理解项目，而不是只按文件树理解。

### 结构

至少覆盖以下章节：

1. 文档目的
2. 背景
3. 模块责任表
4. 多 agent / 多人协作时的边界建议

### 模块覆盖范围

至少包含：

- Home
- Chat
- Cities
- Trips
- Tools
- Admin
- Shared Infra

### 每个模块的说明维度

每个模块应回答：

1. 这个模块当前承担什么角色
2. 它为什么重要
3. 它与哪些模块耦合最强
4. 当前最值得处理的问题是什么
5. 当前最不该顺手处理的问题是什么

## 4. 与现有交接包的联动

2. 在“关键文档索引”中加入这 3 份新文档

### `README.md` 中需要体现的变化

在 `Planning Docs` 中增加：

- `Next 2-4 Weeks Priority Guide`
- `Technical Debt Boundaries`
- `Module Ownership Guide`

## 错误控制与边界

为防止交接材料继续膨胀成重复堆叠文本，本轮遵循以下约束：

1. 每份文档都必须承担单一职责
2. 不在新文档里重复抄写 `HANDOFF.md`
3. 背景故事必须保留，但不能淹没可执行判断
4. 对 agent 的推荐动作必须明确，不能只讲背景
5. 索引必须清晰，不能让后续接手者自己在仓库中搜索

## 验收标准

本轮通过的标准是：

1. 仓库中新增 3 份最后一层交接文档
2. 每份文档都同时包含背景、判断和推荐动作
3. `HANDOFF.md` 中能找到这 3 份文档入口
4. `README.md` 中也能找到这 3 份文档入口
5. 新的 coding agent 在不看聊天记录的情况下，也能回答：
   - 接下来 2-4 周先推进什么
   - 哪些技术债先别碰
   - 不同模块的边界在哪里

## 交付方式

这轮不涉及产品版本升级。  
只会：

- 新增 3 份文档
- 更新 `HANDOFF.md`
- 更新 `README.md`
- 提交并推送到 GitHub
