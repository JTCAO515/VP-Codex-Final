# v2.3：/chat 右侧面板（历史 + 订单）设计文档

日期：2026-05-22  
状态：已与用户确认，待实现

## 0. 目标与非目标

### 目标
1) 在 **同一个 `/chat` 页面**完成历史记录与订单回看：  
   - 对话消息（ChatMessage）  
   - 行程版本（itinerary_versions）  
   - 酒店订单（HotelBooking）  
   - RFP/报价（RFP/Quote）与服务订单（ServiceOrder）
2) 权限与归属正确：所有订单类读写必须 **登录**；后端不再信任前端传 `user_id`，统一从 **Supabase JWT** 解析 `sub`。
3) 不新增页面：不做 `/history`，只扩展 `/chat` 的 **右侧详情面板**（桌面端），移动端以最小可用弹层/抽屉实现。

### 非目标（本迭代不做）
- 复杂订单 UI（卡片、富文本报价、图片附件等）
- 供应商端 UI 大改（supplier portal 继续沿用既有接口）
- 支付/退款
- 多租户/组织

## 2. 前端设计（/chat 右侧面板）

### 2.1 桌面端布局
`/chat` 三栏布局：
- 左侧：历史会话列表（已有）
- 中间：消息流 + 输入框（已有）
- 右侧：**Trip 详情面板（新增）**

右侧面板内容：
- 顶部：Trip 标题 + 更新时间
- Tabs：
  1) **行程**：展示 `itinerary_versions`（版本列表 + 当前版本摘要）
  2) **酒店**：展示本 Trip 的 `hotel_bookings` 列表；支持查看详情与取消（登录态）
  3) **RFP&订单**：展示本 Trip 的 `rfps` 列表 → 选中后展示 quotes 概览；若已生成 service order 则展示订单状态与 fulfillment_info

### 2.2 移动端（最小可用）
- 默认隐藏右侧面板
- 在 header 加一个 “Details” 按钮（或底部栏按钮）
- 点击打开一个全屏弹层（或 bottom sheet）展示与桌面端同样的 tab 内容

### 2.3 游客模式
- 游客可继续聊天与本地历史（最近 3 个会话）
- 右侧面板：
  - 行程 tab：可展示（来自 chat reply / 本地）
  - 酒店 / RFP&订单 tab：若点击“创建/查看”操作，提示登录（toast/dialog）

## 3. 后端 API（新增/改造）

### 3.1 统一鉴权规则
- 订单类写操作、读操作、列表查询：**必须登录**  
  - 使用 `Authorization: Bearer <supabase_access_token>`
  - `user_id` 由后端从 token 的 `sub` 解析得到
- 游客：仅允许 chat（以及必要的 public-config/health）

### 3.2 新增接口（供右侧面板加载）
1) `GET /api/trips/{trip_id}`（登录态）
   - 权限：`trip.user_id == current_user_id`
   - 返回：id/title/updated_at/cities/current_itinerary/itinerary_versions/constraints（如需）

2) `GET /api/trips/{trip_id}/hotel-bookings`（登录态）
   - 权限：通过 Trip 归属校验
   - 返回 bookings 列表（id/status/offer_id/created_at）

3) `GET /api/trips/{trip_id}/rfps`（登录态）
   - 权限：通过 Trip 归属校验（或 rfp.user_id 校验）
   - 返回 rfps 列表（含 service_types/status/created_at）

4) `GET /api/trips/{trip_id}/service-orders`（登录态）
   - 权限：通过 Trip → RFP 归属校验
   - 返回订单列表（id/status/supplier_id/created_at）

### 3.3 改造接口（强制归属、移除信任 user_id）

#### RFP
- `POST /api/rfps`
  - 变更：删除 body 的 `user_id` 字段（或保留但忽略）
  - 归属：`rfp.user_id = current_user_id`
  - 需要登录：是

- `GET /api/rfps/{rfp_id}`
  - 需要登录：是
  - 隔离：`rfp.user_id == current_user_id`

#### ServiceOrder
- `POST /api/service-orders`
  - 需要登录：是
  - 隔离：通过 `rfp_id -> rfp.user_id` 校验

- `GET /api/service-orders/{order_id}`
  - 需要登录：是
  - 隔离：通过 `order.rfp_id -> rfp.user_id` 校验

#### HotelBooking
- `POST /api/hotel/bookings`
  - 需要登录：是
  - 隔离：通过 `trip_id -> trip.user_id` 校验

- `GET /api/hotel/bookings/{booking_id}` / `POST /api/hotel/bookings/{booking_id}:cancel`
  - 需要登录：是
  - 隔离：通过 `booking.trip_id -> trip.user_id` 校验

## 4. 数据模型策略（最小改动）

不强制给每张表新增 `user_id` 字段，采用“**沿链路回溯归属**”：
- HotelBooking → trip_id → Trip.user_id
- ServiceOrder → rfp_id → RFP.user_id
- Quote → rfp_id → RFP.user_id

优点：迁移最小、易落地。  
缺点：查询需要 join/二次查询；后续可优化冗余 user_id 字段。

## 5. 验收标准

1) 登录用户在 `/chat` 右侧面板能看到：
   - 行程版本列表
   - 酒店订单列表（可查看/取消）
   - RFP 列表与 quote 概览，service order 状态
2) 游客仍可聊天；但订单类接口访问均返回 401；UI 有清晰提示引导登录。
3) 后端所有订单/RFP/booking 读写均按 JWT 的 `sub` 做隔离，不存在越权读取。
