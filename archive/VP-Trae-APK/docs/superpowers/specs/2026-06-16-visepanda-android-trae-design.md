# VisePanda Android 接手优化设计

日期：2026-06-16

## 目标

在现有 `VisePanda-Android` 基础上做一轮“最小可用修复版”优化，新仓库名定为 `visepanda-android-trae`。

这一轮不做大规模重构，重点是把安卓端当前最关键的断点打通，让项目从“界面已成型但核心链路未完全闭环”提升为“主要功能真实可用、结构更稳定、便于继续迭代”。

## 范围

本轮修改仅包含以下五类工作：

1. 修复聊天 SSE 协议适配
2. 修复地图接口解析
3. 修复城市列表入口与相关导航
4. 打通 Trip 本地保存闭环
5. 做一轮小范围代码收敛

以下内容明确不在本轮范围内：

- 不重做整体 UI 风格
- 不升级为全新架构
- 不新增推送、离线缓存、iOS 版本
- 不接入新的后端接口
- 不做大规模组件拆分或业务重写

## 当前问题判断

基于已阅读的 `HANDOFF.md`、安卓源码和网页端源码，当前安卓项目的主要问题不是“页面不存在”，而是“若干产品链路和真实后端协议未对齐”。

当前关键缺口如下：

### 聊天链路

网页端 `/api/chat` 的 SSE 事件主体以 `event: message` + JSON payload 形式发送，payload 中再区分 `token`、`split`、`image`、`faq`、`done`。安卓端 `SseClient.kt` 当前却直接按 `event: token`、`event: image` 等类型分支解析，协议不一致，导致聊天流式渲染存在高风险失效。

### 地图链路

网页端 `/api/map` 返回结构是对象映射，`cities` 下按城市名组织经纬度。安卓端 `MapRepository.kt` 当前按数组读取，会导致真实接口数据无法正确进入页面，地图大概率主要依赖本地 fallback 数据。

### 导航链路

安卓端已有 `CityListScreen` 与 `Routes.CITIES`，但首页 “View All Cities” 当前错误地走到了 `city detail("all")`，导致城市总览入口实际上断开。城市详情页和 Trips 空状态页也存在按钮未接导航的问题。

### Trip 链路

`TripRepository.saveTrip()` 已存在，但项目里没有真实调用点。当前 Trips 页面更像只具备“读取和删除”的展示壳，缺少从聊天结果进入本地保存的闭环。

### 代码收敛问题

项目里同时存在 Retrofit 接口定义、`URL(...).readText()` 直连请求、手写 OkHttp SSE 等多套网络访问方式，存在“声明与真实实现并存”的过渡状态。当前不适合一次性重构，但应收敛明显无效或误导性的部分。

## 设计原则

本轮实现遵循以下原则：

1. 先修可用性，再谈架构纯度
2. 尽量沿用当前 Compose + ViewModel + Repository 结构
3. 只做和当前目标强相关的修改
4. 改动应让 handoff、代码和实际行为尽量重新一致
5. 对未来重构保留空间，但不提前重写

## 方案对比

### 方案 A：最小可用修复版

保留现有架构，围绕协议适配、导航修复、Trip 闭环做定点修改。

优点：

- 改动面最可控
- 最适合当前“接手后先恢复可用”的目标
- 可以最快验证真实功能

缺点：

- 会保留部分过渡实现
- 网络层不会在这一轮彻底统一

### 方案 B：网络层统一版

在方案 A 基础上，把普通接口和聊天都收敛到统一的数据访问方式。

优点：

- 结构更整洁
- 后续维护更轻松

缺点：

- 改动面显著扩大
- 首轮交付速度下降
- 更容易引入额外问题

### 方案 C：深度重构版

重新梳理导航、状态、数据层，全面统一安卓端实现。

优点：

- 长期收益最高
- 一次性清掉较多历史包袱

缺点：

- 风险最高
- 不适合当前以“快速接手并恢复主要能力”为目标的阶段

## 选型结论

本轮采用方案 A。

理由很明确：当前最重要的是把聊天、地图、城市入口和 Trip 保存等真实使用链路修通，而不是先追求“完美重构”。只要这些关键链路恢复工作，项目就能回到一个适合继续迭代的状态。

## 具体设计

### 1. 聊天 SSE 修复

目标是让安卓端严格适配网页后端当前的 SSE 格式。

调整方向：

- 修改 `SseClient.kt`
- 以 `event` 字段区分 `message`、`error`、`done`
- 当 `event = message` 时，解析 `data` 对象中的 `token`、`split`、`image`、`faq`
- 保持 `ChatRepository` 与 `ChatViewModel` 的公开接口不变，避免上层 UI 大改

预期结果：

- `ChatScreen` 能正常看到流式 token
- 多 bubble 分段重新生效
- 图片和 FAQ chip 能随流式消息进入 UI
- 错误态仍能通过 `ChatUiState.error` 呈现

### 2. 地图数据修复

目标是让地图优先使用真实 API 数据，而不是长期依赖 fallback。

调整方向：

- 修改 `MapRepository.kt`
- 按 `{ cities: { cityName: { lat, lng } } }` 结构解析
- 必要时根据城市 key 从已有城市数据或 fallback 表中补 `nameCn`、`vibe`、`days`
- 保留 fallback 逻辑，作为网络失败时的兜底

预期结果：

- 地图页加载线上接口时能真正生成 marker
- fallback 只在接口异常时使用
- 地图页行为与 handoff 描述重新一致

### 3. 导航与入口修复

目标是把已经存在的页面真正接入用户路径。

调整方向：

- 首页 “View All Cities” 按钮改为跳转 `Routes.CITIES`
- `CityDetailScreen` 底部 CTA 改为跳转 `Routes.chatCity(cityName)`
- `EmptyTrips()` 中的 “Start Planning” 改为跳转聊天页
- 视需要将导航回调从 `NavGraph` 继续向下透传，避免页面内部写死 TODO

预期结果：

- 首页可以进入完整城市列表
- 城市详情页可以带城市上下文发起聊天
- 空 Trips 页面不再是死按钮

### 4. Trip 保存闭环

目标是让 Trips 页面显示真实保存结果，而不是仅保留空壳。

调整方向：

- 在聊天完成后识别可保存内容并写入 `DataStore`
- 尽量延续网页端“对行程类结果自动保存”的思路，但实现保持克制
- 第一轮只要求实现“保存并展示”
- 不在本轮实现分享、复制、重新载入聊天会话等扩展能力

建议策略：

- 先采用轻量规则判断，例如消息中出现明显的 itinerary / day-by-day 结构时触发保存
- 保存字段使用现有 `Trip` 模型：`title`、`city`、`days`、`content`
- 若规则不足够稳定，可退化为“聊天完成后提供手动保存按钮”，但优先尝试自动保存

首选实现：

- 优先做自动保存
- 如果自动保存实现复杂度超出预期，则改为稳定的手动保存按钮，并确保按钮出现时机明确

本轮成功标准：

- 聊天产出至少能被保存为本地 Trip
- `TripsScreen` 可读取并展示
- 删除功能继续可用

### 5. 小范围代码收敛

目标不是深度重构，而是减少误导和明显失配。

调整方向：

- 清理与真实接口结构明显不一致的代码
- 如某些 Retrofit 声明完全未使用且会造成理解偏差，可删除或留下注释说明
- 尽量把普通 REST 请求收敛到统一风格，但不强行全量改造
- 保持文件命名和模块边界与当前项目一致

预期结果：

- 新接手的人更容易看懂“当前真正生效的代码路径”
- handoff 与代码现状差距缩小

## 数据流变化

### 聊天

用户输入  
→ `ChatViewModel.sendMessage()`  
→ `ChatRepository.streamChat()`  
→ `SseClient` 发起 `/api/chat`  
→ 解析 `message/error/done` 事件  
→ 转换为 `ChatEvent.Token / Split / Image / Faq / Done / Error`  
→ `ChatViewModel` 组装 UI 状态  
→ `ChatScreen` 流式渲染  
→ 聊天结束后触发 Trip 保存判断

### 地图

`MapViewModel.load()`  
→ `MapRepository.getMarkers()`  
→ 解析 `/api/map` 返回对象  
→ 转换为 `MapMarker` 列表  
→ `MapScreen` 生成 marker  
→ 接口异常时退回 fallback 列表

### Trip

聊天完成  
→ 满足保存条件  
→ `TripRepository.saveTrip()`  
→ `DataStore`  
→ `TripsViewModel` 订阅更新  
→ `TripsScreen` 展示最新 trip

## 错误处理

本轮错误处理遵循“继续显示可用结果，必要时优雅降级”的原则。

- 聊天解析失败：显示错误 banner，不使页面崩溃
- SSE 中断：保留已收到内容，结束流状态
- 地图接口失败：继续使用 fallback marker
- Trip 保存失败：不影响聊天显示，但应给出日志或界面提示
- 导航参数为空：采用安全默认值并避免页面崩溃

## 测试与验证

本轮至少验证以下内容：

1. 聊天页可以正常看到流式文字
2. 聊天页可以显示分段、多 bubble、FAQ 或图片事件中的至少一种扩展内容
3. 地图页可以从真实接口解析出 marker
4. 首页可以进入城市列表
5. 城市详情页 CTA 可带城市进入聊天
6. 聊天结果可进入 Trips 页面
7. Trips 删除后界面同步更新

若环境允许，还应补充：

- `assembleDebug` 或 `assembleRelease` 构建验证
- 基础手工 smoke test

## 仓库与交付说明

用户要求新仓库名为 `visepanda-android-trae`。

本轮执行时将按以下方式处理：

1. 先基于原安卓仓库创建新的工作副本
2. 在新副本中完成修改
3. 若当前环境已具备 GitHub 写权限，则创建并推送到同名远程仓库
4. 若当前环境缺少 GitHub 凭据，则先完成本地新仓库整理，并把后续 push 所需步骤明确告知用户

这样可以避免因为外部凭据问题阻塞代码优化本身。

## 不确定项与取舍

当前唯一需要在实现时动态取舍的是 Trip 保存策略：

- 若自动保存规则很容易稳定落地，则优先自动保存
- 若自动判断会导致误保存或实现复杂度明显上升，则退为手动保存

除这一点外，其余设计已足够明确，可以进入实现计划。

## 完成标准

本轮工作完成时，应满足以下结果：

- 新工作副本命名为 `visepanda-android-trae`
- 聊天 SSE 与 web 后端协议对齐
- 地图接口解析正确
- 城市列表入口与关键按钮导航打通
- Trip 保存闭环可用
- 代码中的关键误导点得到收敛
- 项目具备继续迭代的稳定基础
