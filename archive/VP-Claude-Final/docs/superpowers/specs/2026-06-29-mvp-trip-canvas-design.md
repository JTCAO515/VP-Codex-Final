# VisePanda 重做 · Phase 1 MVP 设计:Trip Canvas + Butler Rails

**日期**: 2026-06-29
**状态**: 已通过用户确认,待转入实施计划

## 背景

VisePanda 是面向入境中国的英语国家游客的 AI 旅行助手。此前三个并行版本(Hermes / Codex / Claude)分别探索了不同的产品形态,均收敛到"Python stdlib + Vanilla JS + Vercel"技术栈,功能完整度以 Claude 版最高(真实接入携程联盟、美团联盟、高德地图、Qwen3 语音)。

这次在新仓库 `JTCAO515/VP-Claude-Final` 重新出发,**分阶段**推进,不直接复制任何旧仓库代码。本设计文档覆盖 **Phase 1 MVP**:核心 AI 管家交互页面。

## 范围(Phase 1)

**只做一个页面**:Trip Canvas + Butler Rails 工作台。

不在本阶段范围内(留给后续阶段):
- 城市库、工具箱、我的行程列表等独立 tab
- 登录注册 / 用户账号体系
- 后端数据库持久化(行程数据仅存浏览器 localStorage)
- 真实第三方服务集成(DeepSeek/地图/酒店/翻译等均用占位 key + 优雅降级 mock)
- 背景图随场景/城市动态切换(本阶段背景固定一张,动态切换是已确认的后续阶段需求)

## 目标用户与价值主张(沿用既有定位)

英语母语/英语使用的访华游客,首次或独立出行,25-55 岁,移动端为主。核心价值:用对话代替表单填写规划行程,AI 像管家一样主动补充签证/支付/交通等实操提醒,而不只是回答问题。

## 技术架构

- **框架**: Next.js 14(App Router)+ React + TypeScript,前后端同一代码库
- **状态管理**: Zustand,管理对话历史 + Day 卡片集合 + 管家提醒集合
- **样式**: Tailwind CSS + 自定义 CSS(水墨背景、压暗蒙版效果等无法用 utility class 表达的定制视觉)
- **AI 调用**: Next.js Route Handler `/api/chat`,流式代理 DeepSeek;无 key 时走本地关键词规则 mock,保证演示不依赖真实 API
- **持久化**: 无数据库;Zustand store 通过中间件同步到 localStorage
- **部署**: Vercel(Next.js 原生支持)

**与历史版本的关系**:不复用旧仓库代码(技术栈不同)。后续阶段加城市库/酒店/翻译数据时,会参考 Claude 版的数据结构(`data/cities.json` 等)作为内容设计参考,但代码重写。

## 布局设计

### 桌面端
```
┌─────────────────────────────────────────────┐
│  顶部 Butler Rails 横条(签证/支付/交通/天气/   │
│  风险/语言翻译/紧急事项),常驻,横跨全宽          │
├───────────────────────────┬───────────────────┤
│  左:Trip Canvas            │   右:对话区        │
│  Day 1 / Day 2 / Day 3...  │   持续聊天          │
│  每天含:城市/景点/美食/      │                   │
│  住宿/交通/节奏/预算提示     │                   │
└───────────────────────────┴───────────────────┘
```
背景为固定一张暖调中等浓度水墨山水画(留白占比约 60%),工作台浮于背景之上,左右分栏不用明显容器边框分隔。

### 移动端
默认全屏对话区;Butler Rails 折叠为可横滑的顶部小标签条;右下角浮动按钮"查看行程"以全屏面板唤出 Trip Canvas,关闭后回到对话。画布与对话不同屏共存(对话为主,画布按需查看)。

## 视觉处理:文字直接显示在背景上

不使用半透明对话框/卡片背景。统一采用**局部压暗蒙版**方案:

- **Day 卡片**:每张卡片背后是渐变压暗蒙版(无矩形边框、无卡片背景色),米白色文字叠加在蒙版上,卡片靠间距和蒙版区域自然分隔
- **Butler Rails 条**:同样压暗蒙版处理,图标+短文字
- **对话消息**:用户消息和 AI 消息都不用气泡背景;AI 消息蒙版带稍宽,用户消息蒙版带更窄更淡,靠这个区分发言人而非颜色对比
- **配色**:暖米色/赭石为主背景,朱砂红(`#a23728`)作为强调色(管家提醒里的紧急/风险标记),文字主色米白(`#f7f0e2`)/浅金
- **字体**:标题用衬线/手写感字体(如 Caveat),正文统一系统无衬线字体(保证中英混排可读性)

## AI 数据流与卡片更新机制

1. 用户在对话区发送消息 → 请求 `/api/chat`
2. 后端将对话历史 + 当前行程状态(Day 卡片 JSON + Butler Rails JSON)作为上下文,调用 DeepSeek 流式接口
3. AI 单次回复包含两部分:
   - **聊天正文**:自然语言,流式逐字返回,实时显示在对话区
   - **结构化指令块**:回复末尾附带 JSON,描述本轮对 Day 卡片 / Butler Rails 的增删改操作(例如新增/替换/删除某一天的行程,或更新某条管家提醒)
4. 前端流式渲染聊天文字;流结束后解析末尾 JSON 块,通过 Zustand reducer 增量应用到画布状态,触发新增/替换/重排的过渡动画

## 容错策略

- 结构化 JSON 解析失败或本轮无指令块 → 静默忽略,只更新对话区,画布维持原状,不报错、不打断对话
- DeepSeek key 缺失或调用失败 → 走本地关键词规则 mock(识别城市名/天数等关键信息),生成示例 Day 卡片与回复,保证无真实 key 也能完整演示交互
- 用户表达与现有行程冲突(如行程天数变化)→ 由 AI 在 prompt 约束下自行判断如何重排,前端不做额外冲突校验,只负责渲染最终结构化结果

## 测试范围

MVP 阶段不追求高覆盖率单测,重点验证:
- 结构化指令解析器对残缺/格式错误输入的健壮性(不崩溃,优雅降级)
- mock 模式下端到端可演示(无 key 时完整交互闭环可用)
- 手动走查桌面端 + 移动端布局,确认核心交互闭环可用

不引入 E2E 测试框架(本阶段范围)。
