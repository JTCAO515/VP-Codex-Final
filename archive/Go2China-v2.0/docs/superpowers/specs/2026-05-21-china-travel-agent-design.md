## 1. 背景与目标

面向“外国人来中国旅行”场景，提供一个 Web/手机网页端的对话式 AI Agent：通过少问问题、多交流的方式深挖需求，生成可执行的自由行攻略/行程，并支持完成关键预订闭环（酒店直订），同时为本地旅行社/导游提供接入能力（定制服务/门票/体验以 RFP 报价模式下单）。

### 1.1 业务目标（MVP）
1) **需求深挖 → 行程生成**：对话中持续产出“结构化行程”（可版本迭代与局部重算）。  
2) **酒店可下单（目标B）**：对话内完成“搜索→解释→确认→创建酒店订单→状态同步”。首期允许采用“到店付/担保”等不依赖平台收单的方式。  
3) **定制服务/门票/体验可下单（RFP）**：用户一键生成 RFP → 供应商报价/改方案 → 用户确认生成服务订单（首期不做支付）→ 供应商回传履约信息与状态。  
4) **多语种双向翻译 + 讲解**：文本翻译、图片翻译（OCR）、菜名/景点讲解（知识库 + 检索增强）。  
5) **供应商开放入驻**：首期免费开放入驻（后期可演进为签约供应商与更强的履约/支付能力）。

### 1.2 非目标（首期不做）
* 平台代收代付、分账、强支付能力（首期**不做支付**）。  
* 覆盖机票/火车票的自动出票闭环（可作为后续里程碑）。  
* “AI 自动生成即下单”的固定套餐一口价（风险与纠纷成本高）。  

### 1.3 成功指标（可量化）
* 行程产出：用户从首次进入到拿到可执行行程 v1 的平均轮次/时长。  
* 转化：酒店搜索→下单转化率；RFP 发起→收到报价率→成单率。  
* 满意度：多语种沟通满意度；行程“可执行性”反馈；供应商响应时效。  

## 2. 用户与关键场景

### 2.1 目标用户（入境游客）
* 需要“像顾问一样”的对话式服务：希望少填表，偏好自然交流。  
* 强依赖翻译与解释：菜单、路牌、酒店政策、景点典故与注意事项。  
* 对中国本地出行规则不熟：实名、入住政策、沟通方式、支付习惯差异。

### 2.2 供应商（旅行社/导游）
* 免费开放入驻，能力差异大：既有小团队也可能有技术团队。  
* 需要一个“接单-报价-改方案-履约回传”的最小闭环；后续可签约、佣金、SLA、支付等。

## 3. 总体方案（推荐架构：Orchestrator + 工作流状态机 + 工具/插件）

### 3.1 架构总览
* **Chat Orchestrator（对话大脑）**：意图识别、槽位抽取、决策下一步动作（问/产出/下单/发RFP/翻译）。  
* **Workflow / 状态机（确定性）**：酒店下单、RFP 流转、订单状态变更采用确定性流程，保证可控与可追踪。  
* **Trip Memory（记忆层）**：用户画像 + 本次行程 + 对话摘要 + 版本化行程。  
* **Connectors（工具层）**
  - Trip.com(携程) 酒店 API
  - OCR/图片翻译
  - KB/RAG（菜名/景点/常识）
  - Supplier Hub（RFP/报价/履约）

### 3.2 关键设计原则
* **少问问题，多版本产出**：不追求一次收齐信息；每轮提供“可执行增量”。  
* **只在高价值缺口提问**：缺口影响排程/下单/价格分歧才问；问题尽量用 2~4 个选项。  
* **结构化优先**：行程与订单是结构化对象，不是纯文本，便于后续下单与验证。  
* **可追溯**：所有关键动作（下单、发RFP、供应商改价）有事件日志与版本号。  

## 4. 对话与需求深挖机制（混合型）

### 4.1 每轮对话的固定步骤
1) 识别意图：规划/改行程/酒店下单/RFP/翻译/解释  
2) 槽位抽取：从自然对话提取结构化字段  
3) 缺口评估：距离“可排程/可下单”还差什么  
4) 选择一个主动作：继续交流 / 生成新行程版本 / 轻量追问 / 进入工作流 / 调用翻译工具  

### 4.2 槽位（Slots）分层
**用户画像（长期）**：语言、旅行风格、预算层级、饮食禁忌、住宿偏好  
**本次行程（短期）**：城市、日期/天数、同行人、节奏、必去/不去、通勤阈值、约束（到达/离开时间）  
**下单必填（订单层）**
* 酒店：入住退房、人数/房间、床型/禁烟/含早、联系人信息  
* RFP：服务类型、语言、预算范围、偏好与禁忌、集合/接送要求、时间窗口  

### 4.3 追问策略（Question Policy）
* **硬性缺口（必须问）**：不问无法排程或下单（日期/人数/城市等）。  
* **分歧缺口（选项问）**：不同答案导致路线/价格显著不同（节奏/预算/酒店位置偏好）。  
* **锦上添花（不问）**：先按默认生成 v1，再用推荐引导用户纠偏。  

追问格式：**1 句问题 + 2~4 个选项 + “也可以自己输入”**。每轮最多 1 个追问，避免“审讯感”。

### 4.4 行程版本化（Itinerary Versioning）
* v1 粗排：城市 + 天数 + 每天主题  
* v2 细排：上午/下午/晚上 + POI + 时长估计  
* v3 可执行：交通建议 + 餐饮建议 + 备选方案 + 风险提示  
* v4 可预订：酒店候选、RFP建议、体验/门票候选  

支持“局部重算”：用户只改某天/某偏好 → 只重算受影响的区间。

## 5. 多语种翻译与讲解

### 5.1 总体策略：语义中间层
1) 输入语言检测 → 抽取结构化语义（地点/时间/偏好/意图）  
2) 内部检索与下单使用统一索引语言（建议以中文作为主索引 + 同义词/外文别名）  
3) 输出再翻译为用户目标语言（双向、多语种）  

### 5.2 专有名词保护
地点/酒店/菜名输出统一三段式：
* 原文（中文）  
* 转写（拼音/英文）  
* 解释（是什么、口味/看点、怎么点/怎么玩）  

### 5.3 图片翻译（菜单/路牌/规则）
流程：OCR → 版面结构化（行/列/价格）→ 翻译 → 输出“点餐建议/过敏提示/避坑提示”。  

### 5.4 知识库（KB/RAG）结构
拆分：
* **通用库**：常见菜系/菜名、热门景点、入境旅行常识、礼仪与避坑  
* **本地库（可由供应商补充）**：集合点提示、季节性建议、小众路线、当地规则  

条目模板（建议字段）：
* 标题（多语种/拼音）、一句话简介、亮点、时长/最佳时间、预约要点、注意事项、适合/不适合人群、关键词

## 6. 预订与下单工作流

### 6.1 酒店直订（Trip.com）
目标：对话内完成搜索与下单闭环，并能解释关键政策（取消、入住、实名、押金等）。

**状态机（示例）**
1) collecting_requirements（补齐入住退房/人数/偏好）  
2) searching_offers（调用 API 获取候选）  
3) explaining_and_shortlisting（解释差异、筛选）  
4) confirming_details（入住人/联系人/特殊要求）  
5) creating_booking（创建订单）  
6) confirmed | failed | cancelled（结果与后续查询/取消）  

首期策略：尽量选择不依赖平台收单的产品形态（如到店付/可取消等，具体取决于供应链能力）。

### 6.2 定制服务/门票/体验：RFP 报价模式（首期不做支付）
**用户侧**
1) 生成 RFP：从行程/对话摘要中打包需求  
2) 分发：按城市/语种/标签匹配供应商  
3) 收报价：支持多轮改方案/改价  
4) 用户选择：确认某个报价  
5) 生成服务订单：状态进入 created  

**供应商侧**
* 接收 RFP → 报价/提交方案 → 可 revise → 成交后回传集合信息与履约状态

**服务订单状态（建议）**
created → accepted_by_supplier → in_progress → completed  
以及 cancelled / dispute（后续扩展）

**“可下单”定义（首期）**
* 服务订单创建成功 + 供应商确认接单 + 回传集合/联系人等履约信息  

## 7. 供应商开放入驻（免费）与后台/接口

### 7.1 入驻与风控（MVP）
* **免费开放入驻**：支持供应商自助注册/提交资料。  
* **分级启用**（推荐）：默认“可入驻不可接单”→ 完成资质/资料校验后“可接单”。（首期可人工审核；后续可规则化/自动化）  
* 供应商标签与能力：城市、语种、主题、服务类型、可接单时段、参考价。  
* **后续签约供应商**：在不改变 API/数据模型前提下，引入“签约等级/服务保障/更高曝光/佣金/支付能力”等能力扩展。

### 7.2 接入方式
* **供应商后台（优先落地）**：覆盖多数无技术团队供应商  
* **供应商 API**：供技术型供应商对接（与后台共用同一服务能力）

## 8. 数据模型与 API（高层）

### 8.1 核心实体
* User、Trip、Itinerary（版本化）  
* HotelBooking（含状态）  
* RFP、Quote（多轮）、ServiceOrder（未支付，含履约信息）  

### 8.2 主要 API 组
**用户侧**
* /chat/messages  
* /trips/{id}、/trips/{id}/itinerary:regenerate（支持局部重算参数）  
* /hotel/search、/hotel/bookings、/hotel/bookings/{id}、/hotel/bookings/{id}:cancel  
* /rfps、/rfps/{id}、/rfps/{id}:shortlist、/service-orders  
* /translate/text、/translate/image、/kb/search  

**供应商侧**
* /suppliers/profile、/suppliers/capabilities  
* /supplier/rfps、/supplier/rfps/{id}、/supplier/rfps/{id}/quotes、/supplier/quotes/{id}:revise  
* /supplier/orders、/supplier/orders/{id}/status、/supplier/orders/{id}/fulfillment、/supplier/orders/{id}/messages  

## 9. 权限、合规与风险（MVP最低要求）
* 隐私：个人信息（入住人/联系方式/护照信息）加密存储与访问审计。  
* 内容合规：景点/政策描述避免敏感或不实承诺；对供应商方案与用户沟通做留痕。  
* 纠纷预案：首期不做支付，但要有取消/改期流程与证据链（报价版本、聊天记录、状态事件）。  
* 反滥用：供应商刷单/骚扰、用户恶意提交等基本限流与封禁机制。  

## 10. 可观测性与运营面板（建议）
* 关键漏斗：对话→行程v1→酒店搜索→酒店下单；对话→RFP→报价→成单  
* 供应商响应：RFP 首响时间、报价完成率、成交率、履约完成率  
* 质量：翻译/讲解被“改写/不准确”反馈比例；行程被大幅回滚次数  

## 11. 迭代路线图（建议）
* M0：自由行结构化行程 + 多语种翻译（文本）  
* M1：酒店直订（Trip.com）闭环 + 图片翻译（OCR）  
* M2：RFP 供应商后台 + 供应商开放入驻 + 服务订单状态回传  
* M3：门票/体验 SKU 化（部分标准化）+ 供应商 API 完善  
* M4：支付/签约供应商/佣金结算（在合规与风控准备完成后）  

## 12. 开放问题（需要在实施前确认）
1) Trip.com 开放平台的具体能力范围：是否支持到店付/担保、取消、订单状态回调、入境用户证件字段要求等。  
2) 供应商入驻审核策略：完全自助 vs 人工审核门槛（与风险成本相关）。  
3) 多语种范围：首期明确支持哪些语言（UI、本地化、客服/运营能力）。  
4) 知识库来源与更新机制：通用库的数据来源、供应商本地库的审核与质量控制。
