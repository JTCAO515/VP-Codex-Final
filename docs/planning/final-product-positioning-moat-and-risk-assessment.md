## 1. 最终产品定位

> **VisePanda 是外国人来华旅行的城市执行助手（China Travel Execution Copilot for Foreign Visitors）。**
>
> 帮外国游客在中国把事情做成：能抵达、能支付、能沟通、能找到地方、能去到那里、遇到问题能被救回来。

这个定位由三层构成，三层缺一不可，这是本文档与最早那份"收缩成纯救急工具"的规划书之间**唯一还保留的分歧**（已在第一轮评审中裁定，四模型和操作者均未推翻）：

1. **Survival Layer 生存层**——支付、翻译、交通、地址、紧急、离线。目标：用户在中国不要卡死。
2. **Trusted Discovery Layer 可信发现层**——Explore，不是大众点评，也不是普通攻略，目标是帮游客找到"现在能去、风险低、沟通成本低、适合自己"的地点。**保留原因**：Explore 是团队投入最大、也是产品里唯一有复访/传播潜力的资产，四模型这一轮评审虽然对 Explore 现有交互形态（下拉筛选）提出了尖锐批评，但没有一个模型建议砍掉 Explore 本身，批评的是交互设计不是保留决策。
3. **Execution Layer 执行层**——Chat、Trips、Tools、Explore 产生的结果都必须变成动作，不是回答问题。

一句话判断：**救急建立信任，发现带来复访，执行形成闭环。**

## 2. 本轮评审推翻/修正的关键假设

这是本文档相对于此前所有材料最重要的更新——四模型三轮评审暴露出几个**此前没有人正面质疑过的结构性问题**：

### 2.1 "护城河"框架本身问错了问题（高置信度，三模型独立收敛）

项目现阶段**没有真实付费用户、没有真实支付接入、没有真实运营数据**。在这个前提下讨论"建立六个技术壁垒"，三个模型用几乎相同的比喻描述这件事：DeepSeek"在沙地上打地基"、Qwen"沙滩上建丑陋沙堡后设计防盗系统"、Kimi"用战略文档逃避产品验证"。

**结论**：护城河是 PMF（产品市场契合度）验证之后的事，不是现阶段的工程优先级。当前阶段唯一该做的事是验证"外国人在中国现场卡住的问题，是否严重到需要一个独立 App、且现有替代品无法满足"。

### 2.2 六个"技术壁垒"逐一降级为普通工程实践（高置信度，三模型几乎逐条一致）

| 原壁垒 | 三模型一致判断 | 大公司复刻时间估计 |
|---|---|---|
| Execution Card System | 结构化输出+前端卡片组件，是 Function Calling/Tool Use 的常规工程实践 | 1-2 周到 4-6 周 |
| Foreigner Execution Graph | 本质是 if-else 决策树，不是真正的知识图谱；且当前团队规模不可能把它做深 | 携程/高德有十年级客服数据，起点比 VisePanda 高两个数量级 |
| POI Execution Metadata | 六个里唯一有点数据资产味道，但高德/美团用真实用户行为数据跑统计脚本就能生成，是降维打击 | 1 天到数周 |
| Offline Survival Pack | 技术复杂度为零，大公司不做只是优先级问题不是能力问题 | 数天 |
| Human Task SOP | 没有真实客诉，"沉淀 SOP" 是空中楼阁——被三模型批得最狠的一项 | 不适用（需要先有规模化人工介入） |
| Trust Layer | RAG 引用溯源的行业标配（LangChain/LlamaIndex 都有），是入场券不是护城河 | 不适用（本来就该有） |

**架构师核实补充**：三个模型不知道的是，Show to Local（地址大字卡）和 Payment Wizard 这类功能**已经真实存在于生产代码中**，不是从零开始；`curated_pois` 编辑精选层也已经在用高德 API 逐条查证真实存在性，不是模型以为的"编辑瞎写"。但这不影响它们对"六个壁垒"整体判断的有效性——现状的问题是这些已有能力的**规模和数据积累**，不足以构成真正的护城河，这条批评依然成立。

### 2.3 大公司覆盖风险重新评级（高置信度，三模型独立指出同一个漏项）

三轮评审里最一致的一条修正：**项目方原有的风险分级低估了地图厂商和操作系统层的威胁**。

| 对手 | 项目方原评级 | 重新评级 | 理由 |
|---|---|---|---|
| 高德/百度地图 | 中高 | **极高** | 地图是外国游客现场场景的天然入口，加几个"外宾友好"标签+优化外卡打车绑定，就能吸收大部分产品价值，一个季度内可覆盖核心场景 |
| iOS/Android/华为等操作系统层 | （原方案未列入） | **极高（漏项）** | 下一次系统更新直接内置"境外游客模式/入境模式"，是基础设施级别的釜底抽薪，用户连 App 都不用打开 |
| 微信 | （原方案未单独列入） | **高（漏项）** | 外国人来华必须用微信，官方"外籍人士服务"入口一旦上线是降维打击 |
| Trip.com/携程 | 高 | 高（维持） | TripGenie + 供应链已具备，但更偏"预订前后"，现场执行不是其核心战场 |
| 支付宝/微信支付 | 高 | 中（下调） | 只会持续改善支付体验本身，不会扩展成全行程助手，但会削弱 VisePanda 的核心痛点价值 |
| 大众点评/小红书 | 中 | 低（下调） | 内容基因转型现场执行工具的路径长，短期不构成威胁 |

**假想竞争对手的真实反应**（三模型高度一致）：Trip.com 产品负责人的反应是"有意思但不急，等他们跑出真实转化率再抄或者用流量碾压"；高德产品负责人的反应是"这个我们随时能做，暂时不用管"。**没有一个模型认为当前产品形态会让大公司紧张。**

### 2.4 真正可能的差异化路径（此前所有材料都没有正面提出，本轮三模型独立收敛到同一批方向）

抛开六个技术壁垒，三模型收敛到的、更难被大公司复制的路径，全部**不是技术，而是监管缝隙、关系、和运营模式**：

- **监管/合规缝隙**：酒店涉外资质查询、实名认证辅助、身份验证摩擦——这些是"大厂不愿意碰的脏活累活"，反而可能是真正的护城河。
- **B2B2C 白标**：不直接 to C 跟大厂抢流量，而是卖给入境游地接社、涉外酒店、国际航司/旅行社，作为"中国目的地数字化模块"，获客成本由 B 端承担。
- **极端细分垂直社区**（Kimi 独家提出）：放弃"所有外国游客"，专攻穆斯林游客深度游、无障碍旅行、过境签打卡这类通用大平台不屑做的 1% 人群，靠社区网络效应而不是技术。
- **线下"最后十米"运营**（Qwen 独家提出）：机场/高铁站实体服务台、涉外商户线下分润合作，把线上产品变成线下服务的引流工具。

这些路径的共同前提是需要**真人运营/商务资源**，不是工程侧能独立推进的，本文档只记录方向，不代表已经决定投入。

## 3. 两个已拍板的组织/商业决定（操作者已确认，本文档记录执行边界）

### 3.1 平台优先级对调：iOS 主线，Android 追平

**决定**：新功能从 2026-07-06 起先在 iOS 设计和实现，Android 随后追平（原来是反过来）。

**依据修正**：原始理由"欧美/日韩游客 iPhone 占比高"经三模型压力测试后被认为缺乏数据支撑（没有真实用户分布数据佐证）。架构师提出的替代依据更站得住脚：**这几周 Codex 交付的 iOS PR 质量稳定可信**（TTS v2、Issue #84 P1/P2 三个 PR 均用真实 xcodebuild 编译+测试验证过，分支基底干净，自查清单如实），这是基于真实工程产出的判断，不是人口画像猜测。

**执行边界**：
- 只对**新功能**生效，已有功能（Android 现有实现更成熟的部分）不做倒退式迁移。
- "主线"在双 AI Agent 协作模式下的实际含义：iOS 侧先做详细设计（交互稿/数据模型/验收标准），Codex 实现，验证通过后开一个 Android 平行 Issue 给 Antigravity 追平，禁止 Android 自创命名/字段。
- 三模型一致提醒的风险：这不解决"两端体验分叉"的根本问题，只是把"二等公民"从 iOS 换到 Android。**应对**：每个新功能的 Issue 必须包含跨端一致的字段/命名契约（沿用现有 API_SPEC.md 的做法），而不是让 Android 团队"照着 iOS 截图猜"。
- KMM（Kotlin Multiplatform）被模型提及可以从根本上解决双端分叉，但操作者未决定投入，这是一个**已知但未采纳**的长期选项，不纳入近期任务。

### 3.2 商业化方向：信任漏斗，代付明确否决

**否决**：代付模式（帮外国游客用外卡支付，后端用人民币结算给国内商户赚汇率差）——涉及跨境资金流，现阶段合规风险不可接受，操作者原话"代付一定不行，至少现阶段不行"。

**采纳方向**：大部分功能免费，小部分付费，只有对产品产生深度信任的重度用户，才会被引导向高客单价定制服务。高客单价服务需要品牌信任背书，商业化是信任的下游而不是入口。

**三模型压力测试后的关键修正**：这个漏斗对**一次性游客（7-14天停留）在数学上不成立**——三个模型独立拆解了时间线，指出深度信任建立所需的时间远超单次旅行窗口。**架构师补充的折中判断**：产品的目标用户本来就分两类——一次性游客和短期商务/展会访客（后者可能一年多次往返）。信任漏斗模型**只应该对第二类用户生效**，信任可以跨多次行程累积；对一次性游客，不应该指望他们走到漏斗底部，现阶段只需要把免费层价值做扎实。

**待办**：产品需要一种方式区分"这是用户第几次使用"（哪怕只是简单的登录历史/行程次数统计），才能让这个商业化方向真正落地，而不是对所有用户一视同仁地铺开一个不会转化的漏斗。这是一个需要单独设计的功能点，本文档不展开具体方案。

**被识别出的额外合规暗坑**（三模型给出高度重合的清单，均属于"看似合规实则有真实监管风险"，本轮明确列入不做清单）：
- **外币兑换/多币种钱包**——涉及非人民币计价和跨境结算，属于外汇业务，需要国家外汇管理局牌照。
- **"先收钱后结算给商户/服务方"模式（二清）**——无证二清是央行重点打击对象，罚款可达上亿且可直接关停。
- **代订酒店/机票/门票/导游服务**——可能被认定为无证经营旅行社业务，需要《旅行社业务经营许可证》。
- **处理游客的护照/行程/位置数据**——涉及《数据出境安全评估办法》，尤其如果存储或处理涉及境外服务器/境外团队成员访问。
- **CPS 导购抽佣**（推荐商户拿回扣）——一旦被发现有回扣，信任瞬间破产，且若商户涉及欺诈需承担连带责任。

## 4. 技术架构层面已确认需要处理的真实风险

### 4.1 CanvasPatch（LLM 直出结构化行程 diff）架构风险——已确认待评估，不是假设

三个独立模型在两轮评审里都点名警告：让 LLM 直接吐出结构化 diff 来改行程，在跨模型 failover 时风险极高。**这不是理论担忧**——本轮会话期间，架构师亲手修复过一个真实事故：`butlerPrompt.ts` 的 `normalizeDays` 函数只认 `raw.blocks` 字段，DeepSeek 有时候不遵守 prompt 里"blocks 数组"的格式要求，吐出更接近训练分布的旧结构（`morning`/`afternoon`/`evening`.`activities[]`），导致 AI 生成的详细行程内容被静默丢弃。

**处理方式（操作者已同意）**：不直接动手改架构，先做一份技术方案文档，评估"意图识别+参数提取+传统代码 CRUD"替代"LLM 直出 diff"的可行性、影响面（波及 butler-service 后端 + iOS/Android/Web 三端消费逻辑）。

### 4.2 Show to Local 多场景卡组——规格已定稿

`docs/planning/show-to-local-multi-scene-cards-spec.md` 已完成详细规格：现有地址大字卡扩展为 Address / Allergy / Symptom 三种卡片类型，复用同一套大字+朗读+复制交互组件，过敏/症状文案全部人工预翻译，不经 LLM 现场生成（医疗/过敏场景的翻译准确性是硬边界）。Hotel Check-in Credential 第四种卡片评估后排除在本轮之外（需要先建 Booking 数据模型，范围超出"低成本复用"的初衷）。

## 5. 实现路径（不是照搬 90 天路线图，按 PMF 优先级重新排序）

与 Codex 文档的 Phase 0-6 路线图不同，本文档的路径优先级不是"按功能领域推进"，而是**按"验证成本 vs 战略确定性"重新排序**：

### 第一优先级：低成本、高共识、可立即执行

- Show to Local 多场景卡组（规格已定稿，直接进入 iOS 实现）。
- CanvasPatch 架构评估技术方案文档（评估先行，不直接动手）。
- 把 iOS 主线化决定固化进 AGENTS.md/WORKFLOW.md，明确后续 Issue 拆分方式。
- 补充"不做清单"（第 3.2 节列出的合规暗坑），写入共享文档，防止未来任何 agent 无意中触碰。

### 第二优先级：需要先设计后实现

- Explore 交互形态重构——三模型一致批评现有"9城×5品类+四个下拉筛选"是"编辑思维不是用户思维"，应该改为"位置+时段+场景"驱动的情境化过滤（例如：午饭时间默认只显示餐厅、步行10分钟圈内优先）。这不是砍掉 Explore，是重做交互，需要单独出一版设计方案。
- 用户分层机制（区分一次性游客 vs 重复/商务访客），为信任漏斗商业化落地做前置准备。

### 第三优先级：需要操作者判断是否投入（非工程侧能独立推进）

- 低成本验证 iOS/Android 真实用户设备分布假设（landing page/社交媒体调查/App 内埋点）。
- 监管/合规缝隙、B2B2C 白标、线下运营等差异化路径的商务可行性评估。
- 高客单价定制服务的具体形态与履约方案（本文档明确：说不清楚履约方是谁之前，这个方向和之前的"六个技术壁垒"是同一种自我欺骗，需要操作者亲自想清楚）。

---

## 6. 明确的不做清单（本轮新增，叠加已有清单）

- 不做代付/外币兑换钱包/任何涉及跨境资金结算的功能。
- 不做无证的旅行社业务性质的酒店/机票/门票代订交易闭环。
- 不把六个"技术壁垒"当作现阶段的工程优先级去主动投入建设。
- 不在没有真实用户分布数据前，对已有功能做倒退式跨平台迁移。
- 不让 Allergy/Symptom 类卡片的中文文案经过 LLM 现场生成——必须是人工预翻译的静态文案。
- 不对一次性游客铺开"信任漏斗"商业化叙事——现阶段这类用户只需要把免费层做扎实。
