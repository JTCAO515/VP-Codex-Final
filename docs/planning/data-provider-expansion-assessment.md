# VisePanda — 景点/餐饮/酒店数据与预订服务拓展评估(v0.2.17,纯规划)

> 研究方式说明(诚实披露):本轮对六个候选服务(Trip.com、大众点评、百度地图、
> 腾讯位置服务、Klook、KKday)的官方开放平台/文档页面直接 `WebFetch` 均返回
> `403 Forbidden`(这些站点对自动化抓取有反爬限制,不是本环境代理故障——已
> 用 `curl "$HTTPS_PROXY/__agentproxy/status"` 核实代理本身无失败记录)。因此
> 本文结论基于多次独立 `WebSearch` 检索交叉印证官方页面标题/摘要得出,**不是
> 对 FlyAI 那样的逐字源码级核实**。凡标注「⚠️ 待人工核实」的条目,在真正启动
> 对接前必须由人登录官网/开发者控制台确认最新条款,不能直接当作实现依据。
>
> 本轮**不改产品运行时代码**,不新增任何 API 调用、mock 数据或 provider 抽象层。

---

## 背景:这是在补什么缺口

高德地图(Amap)目前是 Explore 唯一的实时数据源,解决的是「这个地方在哪、评分
多少、电话是什么」——**地图 POI 层**的问题。但 VisePanda 面向的是**外国自由行
游客**,他们真正卡住的地方往往不是"找不到地方",而是:

1. **订不到**——找到了景点/酒店,但没有一条路径能在 App 内直接用境外信用卡付款
   下单,只能跳出去自己另想办法(签证要求、当地平台不支持外国手机号注册等)。
2. **看不懂**——大众点评/美团的内容质量最高,但语言和排版是给中国本地用户的。

所以筛选候选服务的核心标准是:**这个服务离"外国游客能直接付款拿到票/房间"
有多近**，而不是单纯"数据全不全"。

## 六个候选服务逐一评估

### 1. Klook(客路)—— 门票/活动,外国游客定位最精准

- **定位**:总部香港,产品从创立起就是给境外/跨境游客用的英文平台,天然覆盖
  "外国人在中国境内买门票"这个场景,不需要额外做翻译/本地化适配层。
- **接入方式分两级**:
  - **Affiliate(联盟)**:门槛低,注册后即可拿到 deep link / 搜索框 widget 挂
    在自己站点上,佣金 5%–8%,但只能跳转外链,**不能拿结构化数据在自己 App
    内原生渲染**——不适合 Canvas 内联展示的产品形态。
  - **Partner API**:面向"旅游科技平台/渠道管理系统"的正式商务对接,能拿到
    产品/库存/价格的实时接口,在自己品牌下完成下单流程,佣金 5%–12%(按品类
    和量级浮动),集成周期约 4–12 周(含审核认证)。⚠️ 待人工核实最新费率与
    申请入口(`klook.gitbook.io/openapi`、`affiliate.klook.com`)。
- **对 VisePanda 的价值**:如果拿到 Partner API,可以直接补上 Canvas Day
  详情里 `food`/`transport` 之外一直缺失的「门票怎么买」这一环,且是六个
  候选里**语言/支付摩擦最小**的一个。

### 2. KKday —— 门票/活动,Klook 的直接对标

- **定位**:总部台北,同样是面向境外游客的活动/门票平台,与 Klook 高度同质化
  竞品,产品覆盖有重叠但不完全相同(常见于两家分别拿到不同独家供应商)。
- **接入方式**:
  - **KKpartners 联盟网络**:面向媒体站/品牌/航司等,拿唯一 CID 追踪码做返佣
    跳转,同样是外链模式,佣金结构随不同渠道浮动(公开资料里从 2%–8% 不等,
    含限时加成)。
  - **B2D(Business-to-Distributor)平台**(`b2d.kkday.com`,部分能力经由
    `rezio.io` 渠道管理系统):面向正式分销商,先在 SIT 测试环境获批,再拿
    API token,可做搜索/选品/下单全流程,白标。门槛看起来比 Klook Partner
    API 更明确写出"先审核成为 Distributor",⚠️ 待人工核实具体商务条件与是否
    对小型/早期产品开放。
- **对 VisePanda 的价值**:与 Klook 定位几乎一致,建议**只接一家先跑通**,
  不必两家同时对接——重复建设两套几乎等价的 provider 抽象收益有限,若要选
  优先做 Klook(公开资料里 API 文档更完整、集成路径更清晰）。

### 3. Trip.com(携程国际版)—— 酒店预订,国际化程度最高

- **定位**:携程旗下面向国际用户的品牌,英文界面、多币种、境外信用卡支付都
  是原生能力,这正是 Canvas Day 详情里 `stay` 字段目前只是 AI 生成文字、
  完全不可预订的缺口。
- **接入方式**:官方开放平台 `connect.trip.com`,公开资料显示至少有三类
  合作形态——酒店库存对接(面向 PMS/渠道管理系统/中央预订系统等"连接性
  合作伙伴")、门票活动 Partner API、以及按量付费的定制合约接口。⚠️ 这三类
  面向的合作方类型不同,哪一类对"内容型旅行 App 想内嵌预订"开放、门槛与
  周期如何,官网页面被反爬拦截未能逐字核实,需要人工登录 `connect.trip.com`
  确认具体准入路径(大概率也是商务审核制,不是自助注册即用)。
- **对 VisePanda 的价值**:六个候选里**唯一能补上"酒店"这个品类**,和
  Klook/KKday(门票)互补而非竞争,应优先于门票类目立项——因为酒店是
  Canvas 六维完成度里权重最高的一环之一。

### 4. 大众点评(美团)开放平台 —— 餐饮内容质量最高,但集成门槛也最高

- **定位**:中文本地生活内容的事实标准,美食点评的详实程度远超高德/百度/
  腾讯地图的基础 POI 信息。
- **接入方式与限制**(多来源交叉确认,一致性较高):
  1. 注册开发者 → 拿测试 App Key → 提交应用审核 → 通过后转正式数据,流程
     本身对个人开发者不设硬性资质门槛。
  2. 但**使用条款明确限制**:不得批量抓取/储存 API 数据,且**明确写有竞品
     条款**——如果产品涉及"用户点评、团购、电子优惠券、商户预订"等与大众
     点评存在竞争关系的功能,平台有权终止协议。VisePanda 的 Canvas 本质上
     会展示"景点/餐饮推荐 + 引导预订",存在被判定为竞品功能的风险,⚠️ 需要
     人工逐条核对最新服务条款(`open.dianping.com`)后再决定是否申请,不能
     假设默认能过审。
  3. 内容语言是中文,即使拿到 API,仍需要额外的翻译/摘要层才能给英文用户用
     ——这是六个候选里唯一需要**额外本地化处理成本**的一项。
- **对 VisePanda 的价值**:内容质量确实是同类最高,但"集成门槛高 + 竞品条款
  风险 + 需要翻译层"三项叠加,ROI 在六个候选里排最后。建议**暂缓**,等产品
  形态和商业模式更成熟、有专职人力走审核流程时再重新评估。

### 5. 百度地图开放平台 —— 高德的同级备选,边际收益有限

- **定位**:与高德地图同属"地图服务商基础 POI"这一层,不是新品类,是备选/
  容灾数据源。
- **接入方式**:Web 服务 API(地点检索 v3 等)对个人开发者免费开放,注册后
  创建应用即可拿 AK(Key),企业认证可以提高调用配额;地点检索返回字段包含
  名称、坐标、分类 tag、评分、营业时间、电话等,和高德的字段集高度重合。
  ⚠️ 免费额度具体数值(公开资料提到"地点检索/建议查询类日限 10 万次"级别）
  需要人工登录控制台核实当前政策,官方页面可能已更新。
- **对 VisePanda 的价值**:除非未来发现高德在某些城市/品类存在明显数据
  盲区,否则新增一条数据重合度极高的 provider 只会增加维护成本(两套 provider
  抽象、两个 key、两套字段映射),不产生新的用户价值。**不建议现阶段接入**,
  仅作为高德服务不可用时的技术容灾选项记录在案。

### 6. 腾讯位置服务(WebService API)—— 同上,备选中的备选

- **定位**:与百度地图类似,是国内地图三强之一,WebService API 面向开发者
  开放地点搜索、逆地理编码、路线规划等能力。
- **接入方式**:注册腾讯云账号 → 创建应用/Key → 按需申请调用配额,配额审批
  官方资料写的是"3 个工作日内",区分个人/企业开发者的额度上限。⚠️ 具体额度
  数值与国际地点覆盖情况需人工核实(有独立的"海外地点搜索"文档入口,提示
  海外覆盖可能是单独产品线,不一定包含在基础额度内)。
- **对 VisePanda 的价值**:结论与百度地图一致——**不建议现阶段接入**,数据
  重合度高、无新增用户价值,仅作技术容灾备选记录。

## 综合优先级矩阵

| 服务 | 补的缺口 | 外国游客友好度 | 接入门槛 | 与现有架构契合度 | 建议 |
|---|---|---|---|---|---|
| **Trip.com** | 酒店预订(Canvas `stay` 字段) | 高(英文+境外卡原生) | 中高,需商务审核,⚠️路径待核实 | 高,复用既有 provider 抽象模式 | **优先研究第一位** |
| **Klook** | 门票/活动预订 | 高(境外游客定位) | 中,Partner API 需商务审核,Affiliate 门槛低但拿不到结构化数据 | 高 | **优先研究第二位** |
| **KKday** | 门票/活动预订(与 Klook 重叠) | 高 | 中高,B2D 需先获批 Distributor | 高,但与 Klook 功能重复 | 仅作 Klook 的备选/补充,不必两家同时接 |
| **大众点评** | 餐饮内容深度 | 低(纯中文,需翻译层) | 高(竞品条款风险,需人工核对) | 中,需额外翻译层 | 暂缓,风险评估优先于技术评估 |
| **百度地图** | 无新增(与高德重合) | 中性 | 低(个人开发者免费) | 高但无增量价值 | 不建议,仅记录为容灾选项 |
| **腾讯位置服务** | 无新增(与高德重合) | 中性 | 低到中(配额需审批) | 高但无增量价值 | 不建议,仅记录为容灾选项 |

**建议的研究/立项顺序**:Trip.com(酒店)→ Klook(门票)→（视 Klook 结果决定是否需要
KKday 兜底)→ 大众点评(需先由人评估竞品条款法律/商务风险,再决定是否值得投入技术
调研)。百度地图与腾讯位置服务不建议投入研究精力,除非高德出现明确的数据缺口。

## 与 `mock-inventory.md` 的关系

现有 `mock-inventory.md` 第 23 项已记录"真实预订数据(飞猪)"为 🔴(架构现实
判断:开发者/Agent 工具,非可直接生产集成的公开 API,需官方合作)。本轮评估
的六个新候选服务性质与飞猪类似——**全部需要官方商务合作或至少通过人工审核**
才能拿到生产可用的数据/预订能力,没有一个是"配置 key 即可用"的自助式 API
(百度/腾讯地图的基础 POI 检索例外,是自助式的,但价值有限)。

因此本轮**不新增** mock-inventory 条目(六个候选目前都停留在"值不值得研究"
的规划阶段,不是"代码已就绪等 key"的状态),待某一具体服务(优先 Trip.com 或
Klook)进入官方合作申请阶段、产出类似 `flyai-skill-integration.md` 的逐字
核实技术画像后,再新增对应条目并标注 provider 抽象设计。

## 下一步(不在本轮执行,供操作者决策)

1. **Trip.com**:操作者或团队需要实际注册 `connect.trip.com` 开发者账号,
   人工确认哪一类合作(酒店连接性合作伙伴 / 定制合约 API)对当前产品规模
   开放,以及最低申请门槛(是否要求已有实际交易量、企业营业执照等)。
2. **Klook**:注册 `affiliate.klook.com` 快速确认联盟计划门槛(几乎肯定
   可以自助通过),同时另行联系 Klook Partner API 团队询问 Partner API 的
   申请条件,评估是否值得为拿到结构化数据走商务流程。
3. **大众点评**:建议**先做法律/商务风险评估**(产品的"推荐+引导预订"
   模式是否会被认定为竞品),而不是先做技术调研——技术上申请流程本身不难,
   风险在条款层面。
4. 百度地图/腾讯位置服务:不建议投入下一步资源,除非未来 Explore 页面在
   实际使用中发现高德在某类城市/POI 上有系统性数据缺口。

一旦操作者选定要推进的服务,下一轮应产出与 `flyai-skill-integration.md`
同等严谨度的逐字技术核实文档(需要实际登录官网/开发者后台获取一手资料,
而不是像本轮这样依赖搜索引擎摘要),再决定是否新增 provider 抽象与
`mock-inventory.md` 条目。
