## 0. 方法论说明(诚实披露)

操作者原本要求引入 DeepSeek v4 Pro、Kimi k2.6、DashScope Qwen3.7 Plus、Zhipu GLM 5.2 四个外部模型对规划书做对抗性评审，并提供了四个模型的真实 API Key。实际执行时发现：**本次会话所在环境的出站网络策略屏蔽了这四个域名**(`api.deepseek.com`、`dashscope.aliyuncs.com`、`open.bigmodel.cn`、`api.moonshot.cn`，代理网关返回 403 策略拒绝，非临时故障)。按环境规则，这类拒绝不能绕过，只能如实报告；操作者确认"先用可以用的方式跑一轮"。

因此本报告的对抗性评审**由架构师(Claude Code)本人以严格自我对抗的方式完成**：先做五角度独立批判(第 1 轮)，再对第一轮的每条结论做反诘、找漏洞、给出可执行的收敛决定(第 2 轮)。所有结论都锚定在仓库当前真实实现状态(而非规划书或架构师的设想)之上核实过。**这不等价于四个外部大模型的真实输出**，如果之后网络策略放开，应重新用同一套背景资料对四个模型发起调用，并与本报告做交叉验证；在此之前，本报告的结论应被视为"单一评审人的严格自我审查"，置信度低于真正的多模型对抗。

## 1. 核心发现:规划书对项目现状的理解已经过时

规划书写作时假设的是一个"Web 优先、Chat/Trips/Explore/Tools/Me 五宫格刚起步"的项目。但仓库当前真实状态(2026-07-06，`package.json` v0.3.22)是：

- 研发主线已从 Web 转向**原生 Android(主线)+ iOS(对齐)双端**，Web 自 v0.2.17 起冻结，只做维护性改动。
- 规划书里列为"待建"的多项能力**已经上线并经过真机验证**：Payment Wizard(交互式支付向导)、Show to Local(中文地址大字卡 + TTS + 复制)、Visa Checker(按国籍/停留天数算免签)、Translate(7 语言 + 后端 TTS + OCR + STT)、Trips 的 Now/Next/Later 时间线 + 基于真实数据的 Readiness 计算、Chat 内工具卡片可点击深链、AI Profile 长期记忆(查看/删除)。
- 规划书主张把 **Explore"降级"为附近可行动地点层、不做内容社区**，但团队恰好反向投入了大量真实工程，把 Explore 做成了一个类大众点评的完整发现层：9 城 × 5 品类、真实定位、四个下拉筛选栏、高德实时 POI + 自建 `curated_pois` 编辑精选层(LLM 生成初稿逐条经高德 API 查证存在性，标注"VisePanda Editorial"，绝不冒充真实用户评价)。**这是规划书与仓库现状冲突最尖锐的一点**，本报告第 3 节正面处理这个冲突。
- 规划书说对的真实缺口：订阅是纯 UI 占位、未接真实支付(StoreKit/Google Play Billing，卡在 App Store Connect 商品 ID 未定)；没有真正的离线可用能力；没有 Emergency 附近医院/警察数据面板；没有专门的 Food Safety 流程；生产环境 butler 记忆接口有反向代理配置缺口。
- 仓库里没有找到独立的"Codex 产品定位想法"文档 —— 在这个项目的协作规则里，Codex 是纯执行角色(按 Issue 开发 iOS/复杂后端逻辑)，无权设定产品定位；定位决策权唯一属于架构师(Claude Code)。

## 2. 五角度对抗性评审(第一轮，摘要)

完整逐条记录见本轮工作产出(未入库，仅内部工作记录)；核心结论摘要：

| 角度 | 关键判断 |
|---|---|
| 外国游客用户 | 规划书说对了支付/地址/翻译/签证是真痛点；但完全没处理"落地前没网络怎么发现并装上 App"的鸡生蛋问题，也漏掉了外卡 ATM 取现这个比支付宝/微信支付失败更早发生的痛点；"小费文化解释"是套用通用旅行 App 模板的伪需求(中国没有小费文化)。 |
| 潜在商业合作伙伴 | 高德已有可升级的合作资产(官方 MCP Server 已验证可用);支付宝外卡受理/云闪付国际卡绑定是当前真实的政策热点，规划书完全没提，可能是最大的合作机会；eSIM/运营商分销直接命中"落地无网络"死结；酒店 2B 前台工具、会展 B2B2C 分发渠道也是规划书未覆盖的机会。 |
| 付费用户/商业化 | 现有月费订阅($9.99/$19.99)与游客几天到两三周的实际停留时长结构性错配，规划书的 Trip Pass(按行程时长)方向正确但单一固定期限缺乏复购抓手；"5 分钟现场任务完成率"是体验指标不是商业指标，不能替代商业验证。 |
| 研发团队/工程组织 | 90 天路线图按 Web 单端语言写成，落地到"Web 冻结 + Android 主线 + iOS 对齐"的真实组织需要重新拆分；历史上已多次因"重新定位/重做"造成返工(Explore 重做 B/C/D/G、版本号顺延)，此次收缩定位如果不处理"沉没成本"问题，容易把刚验证完成的高成本产出当错误推翻。 |
| 未来可扩展性 | 纯"救急"定位天然低频、留存钩子弱、复访理由弱；Explore(发现层)恰恰是当前产品里唯一有复访/传播潜力的部分，规划书建议削弱它是在废弃最有长期增长潜力的资产。 |

## 3. 第二轮自我对抗蒸馏 —— 最终收敛决定

对第一轮结论逐条反诘后，收敛为以下可执行决定：

### 3.1 一句话定位(修订版)

> **"救急内核 + 可信发现层"的中国旅行随身管家** —— 救急能力(支付/翻译/签证/交通/地址/紧急联系)必须做到不能掉线，发现层(Explore)是让用户愿意反复打开、形成口碑的理由，两者缺一不可。既不是规划书主张的"收缩成救急"，也不是继续无边界地"做大而全"。

不采纳规划书"Explore 应降级为纯动作层"的主张：Explore 已完成的 9 城 curated 知识库、四筛选栏、实时定位是团队近期投入最大的真实工程之一，且是当前产品里唯一具备复访/分享/口碑传播潜力的部分，削弱它等于放弃长期增长曲线换取短期叙事整洁。

### 3.2 唯一应立刻排期的新增高优先级项:离线可用能力(收窄范围版)

这是"规划书判断对、仓库代码里没做"的少数真实缺口之一，应予采纳，但收窄到最小可行范围，避免重蹈"90 天路线图按 Web 语言写、落地时爆炸成几十个 Issue"的覆辙：

断网时仍可查看的最小集合(全部是**已经存在于本地状态里的数据**，只是没有暴露成明确的离线可用状态)：
1. 今日行程(已在 Room/本地缓存中)
2. 酒店/当前地点地址(Trip 数据已含)
3. 常用短语(Translate 短语本已是本地静态数据)
4. 紧急联系人(需新增一个本地字段，工作量小)
5. 已生成的 Show to Local 地址卡(本地渲染，不依赖网络)

不采纳规划书列出的全部九类离线数据(含汇率、护照遗失完整 SOP 等)，这些留待离线核心验证后再评估增量价值。

### 3.3 商业化调整:计费单位从"月"改为"次/行程时长"，不新增命名体系

- 现有 `Human Service`(占位 $9.99/月)→ 改为**按次人工任务**计费(数量级不变，单位从 `/month` 改为 `/task`)。
- 现有 `Premium Service`(占位 $19.99/月)→ 改为 **Trip Pass**(单笔行程通行证，建议 7 天/14 天两档，一次性定价，不做自动续订)。
- 理由：月费自动续订在游客离境后仍扣费，是差评/退款风险，也是 App Store 审核对"一次性场景产品做订阅"的常见驳回理由。
- 不新增第三套命名体系，只调整现有 placeholder 的计费周期字段；真实 StoreKit/Google Play Billing 接入仍需等待 App Store Connect 商品 ID 定稿，不在本轮范围。
- 不设立独立的"商业北极星指标"：在真实支付接入之前，任何转化率指标都是噪声。现阶段唯一应该做实的是产品体验北极星 —— "5 分钟现场任务完成率"的埋点(规划书 P0-04 提出过，但仓库里没有找到对应埋点代码，是目前唯一"规划书说了、代码里查无此功能"的关键缺口，应该补上)。

### 3.4 商务合作优先级排序(按"技术就绪度 × 命中程度")

1. **高德正式合作** — 技术上几乎零增量成本(MCP 已验证可用，`curated_pois` 已在天天查证高德数据)，优先级最高。
2. **eSIM/运营商分销** — 直接命中"落地无网络"这个规划书自己反复强调的安全底座缺口，现有 Tools 的 Network 分类已有挂载位置，只需要把内容从"指南"升级为"可分销/联登"。
3. **支付宝外卡受理/云闪付合作** — 命中最大痛点，但涉及金融合规和大厂对接周期，需要操作者出面谈判，非工程侧能独立推进；产品侧现在能做的只是把 Payment Wizard 做得足够扎实，为未来合作铺垫。
4. **酒店 2B 前台工具** — 有效但非紧急，Show to Local 卡片已是现成的最小可行版本，优先级低于前两项。
5. **会展 B2B2C 分发渠道** — 获客价值最高，但完全是商务/市场动作，可与研发并行推进，不消耗研发资源。

### 3.5 "落地前发现/安装"的鸡生蛋问题

这本质是分发/市场问题，不是产品功能问题，不应该被误判为"要开发一个新功能"。产品侧唯一该做的动作：确保首次冷启动流程在弱网/无网环境下也能跑通引导(不因为请求失败白屏或卡死)，分发本身(登机前 Wi-Fi 门户、机场/口岸二维码等)交给运营/商务侧解决。

### 3.6 明确不采纳规划书的部分

- Explore"降级"为纯动作层的主张(第 3.1 节已说明理由)。
- 90 天路线图的阶段划分方式(按 Web 单端语言写成，需要按"Web 冻结 + Android 主线 + iOS 对齐"的真实工程组织重新拆分，不能照搬)。
- "小费文化解释"等生搬硬套的通用境外旅行 App 内容(中国没有小费文化，写进产品是暴露对本地场景理解不足)。
- 规划书第 5 阶段提出的"Trust Layer"(`source_refs`/CanvasPatch Gate/失败诚实化)不是新工作，是现有架构(CanvasPatch、InlineToolCard、`exploreRefs`、`curated_pois` 诚实披露)的既有模式，应定性为"补文档，非重新发明"。

### 3.7 需要操作者拍板、非工程侧能独立决定的事项

- 是否投入商务资源接洽支付宝外卡合作 / 会展分发渠道 / 酒店 2B 合作。
- App Store Connect 商品 ID 何时定稿，决定真实订阅(StoreKit/Play Billing)何时能接入。
