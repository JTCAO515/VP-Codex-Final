# CanvasPatch 架构评估：要不要从"LLM 直出结构化 diff"改成"意图识别 + 代码 CRUD"

版本：v1（评估文档，不改动代码）
日期：2026-07-07
性质：技术方案评估，供操作者决定是否排期，不直接实施

## 0. 结论先行

**建议：不做大改，做局部加固。** 现有架构比三个外部模型批评时想象的更健壮——`parseButlerPatch`/`normalizeDays` 已经是一层独立于各 provider 的归一化+校验层，不是"LLM 输出直接进客户端"。真正的问题不是"没有防御层"，而是"防御层目前是遇到一次线上事故才补一次规则"（reactive），没有一种机制能提前发现新的 provider 格式漂移。本文档建议保留现有架构骨架，把"事后补丁"改成"主动契约校验+可观测性"，而不是推翻重来。

## 1. 现状（已核实，不是猜测）

代码路径：`app/api/chat/route.ts` → `lib/ai/orchestrator.ts` → `lib/ai/butlerPrompt.ts`。

- **不是简单 failover，是并发赛跑**：`orchestrator.ts` 同时向多个已配置的 provider（DeepSeek/Kimi/Qwen/GLM）发起请求，`Promise.race` 谁先返回有效解析结果就用谁的，用一个内存态"熔断器"（`providerHealth` Map）跳过连续失败两次的 provider，避免每次都重新对一个正在故障的 provider 计时超时。
- **已经有归一化层**：每个 provider 的原始 completion 会先过 `parseButlerPatch`（内部调用 `normalizeDays`/`normalizeAssistantResponse`/`normalizeToolCards`），这一层的职责就是把"模型自由发挥的 JSON 形状"收敛成 `CanvasPatch`/`TripDay` 这个客户端契约类型，不合规的字段会被丢弃或给默认值，不会直接把 provider 的原始结构透传给客户端（`normalizeDays` 明确用 `as unknown as TripDay` 只挑出契约里定义的字段，不做 `...raw` 展开）。
- **真实发生过的事故**：`normalizeDays` 最初只认 `raw.blocks` 数组，DeepSeek 有时候不遵守 prompt 里"blocks 数组"的格式要求，吐出更接近训练分布的旧结构（`morning`/`afternoon`/`evening`.`activities[]`），归一化层当时不认识这个形状，只会把它当成空 `blocks: []` 处理——不是解码报错（那样至少会被 `Promise.race` 判定为这个 provider 失败，换下一个赢），而是"看起来成功但内容是空的"，这是最隐蔽的一类 bug。修复方式是新增 `extractBlocksFromLegacyPeriods` 函数识别并转换这种旧形状。

**结论**：架构本身已经预留了归一化层这道防线，这次事故不是"防线不存在"，是"防线只认识已知的漂移模式，遇到新的漂移模式时不报错、只是静默丢内容"。

## 2. 三个外部模型的批评，哪些成立、哪些需要修正

三个模型（DeepSeek/Kimi/Qwen）在两轮评审里都警告"LLM 直出结构化 diff"有跨模型 JSON 漂移风险，这个大方向判断是对的，本次真实事故就是证据。但它们给出的具体描述（"LLM 直接碰底层数据结构""没有校验层直接进客户端"）和实际代码有出入——现有架构已经有校验层，只是这层校验目前是**穷举已知形状**（allowlist 式），不是**契约驱动的强校验**（schema 式，遇到不认识的形状主动报错而不是静默降级）。

## 3. 备选方案对比

### 方案 A：保留现状，只加固归一化层（推荐）

不改变"LLM 生成结构化 patch + 并发赛跑 + 归一化"这套架构骨架，只做两处加固：

1. **把"静默降级"改成"主动报错+可观测"**：`normalizeDays` 目前遇到不认识的形状会返回空 `blocks: []`，改成——如果一个 day 的 blocks 数组解析后为空但原始数据里能看出有内容痕迹（比如 `morning`/`afternoon`/`evening` 等已知非契约字段非空），记一条结构化日志（provider id + 原始字段名列表），而不是静默吞掉。这样下次出现新的漂移模式，能第一时间在日志里看到，不用等用户反馈"行程是空的"才发现。
2. **给 `parseButlerPatch` 加单元测试矩阵**：针对每个已接入的 provider，维护一组"该 provider 历史上真实吐出过的畸形 JSON 样本"作为回归测试用例（`tests/butlerPrompt.test.ts` 已经有雏形，是这次修 bug 时新增的 3 个测试，建议扩展覆盖面而不是只测这一次修的这个漂移）。

**成本**：小，不影响现有功能，不需要改动客户端消费逻辑。

### 方案 B：改成"意图识别 + 参数提取 + 传统代码 CRUD"（三个模型建议的方向）

不让 LLM 直接生成完整的 `TripDay[]`/`blocks[]` 结构，而是让 LLM 只输出一个更小的、更容易校验的"意图 + 参数"结构（例如 `{ intent: "add_block", day: 2, time: "afternoon", title: "...", description: "..." }`），由后端代码（不是 LLM）执行实际的行程数组增删改，保证结构完整性由代码逻辑保证，不依赖 LLM 输出格式的稳定性。

**优点**：从根本上消除"LLM 吐出的整个 days 数组结构错误"这类风险——因为 LLM 不再负责生成整个数组，只负责生成一个动作指令。

**成本**：
- 影响面大：现有 `CanvasPatch` 契约（`app/api/chat/route.ts` 消费方、iOS/Android/Web 三端渲染逻辑）都是围绕"LLM 一次性生成完整 patch"设计的，改成"意图+代码执行"需要重新设计整个契约和三端的行程更新逻辑，不是一个局部改动。
- 新的复杂度：需要把"用户说的话"到"结构化意图"这一步的识别做扎实，这一步本身也可能有 LLM 理解偏差（意图分类错误、参数提取遗漏），只是把风险从"格式对不对"转移到"意图分对不对"，不是零风险，只是风险性质变了。
- 现有并发赛跑机制（多 provider 同时跑，谁先返回有效结果用谁的）在"意图+代码执行"模式下依然需要，复杂度并没有减少多少。

**结论**：这个方向理论上更稳健，但改造成本远超"当前架构下打的一个补丁"，且需要重新设计三端消费逻辑，不建议现在做。

## 4. 建议

**现阶段采纳方案 A（局部加固），不采纳方案 B（架构重做）。**

理由：
1. 本次事故的根因是"归一化层不认识一种新形状"，不是"架构设计缺了归一化层这道防线"——加固现有防线的成本远低于重做架构。
2. 项目现在还没有真实付费用户和真实运营数据（见 `docs/planning/final-product-positioning-moat-and-risk-assessment.md` 第 2.1 节），在这个阶段投入大改造 CanvasPatch 架构的工程资源，优先级应该让位于产品验证类工作。
3. 方案 B 不是不能做，是"现在做的性价比不够"——如果未来 CanvasPatch 相关的静默丢数据事故反复出现（哪怕加固了归一化层之后还是防不住新的漂移模式），说明问题比预想的更结构性，那时候再启动方案 B 的评估。

## 5. 如果要做，分几个阶段（仅供未来参考，本轮不排期）

1. 阶段 1：归一化层加固（本文档方案 A），成本小，可以随时排期。
2. 阶段 2（触发条件：阶段 1 上线后仍然反复出现新的漂移事故）：给 `CanvasPatch` 契约补一层 JSON Schema 校验（比如用 zod），让"不符合契约"在归一化层就是显式报错而不是尽力而为地拼凑，配合现有的 provider 赛跑机制，让格式错误的 provider 直接判定为这次请求失败，换下一个 provider。
3. 阶段 3（触发条件：阶段 2 之后判断"LLM 直出完整 diff"这个模式本身就是问题根源）：启动方案 B 的正式设计，这时候需要单独立项，涉及后端 + iOS + Android + Web 四端协同改造，不是一次 Issue 能完成的工作量。
