# VisePanda — Product Expansion Plan (v0.1.46)

> **Documentation-only strategic planning iteration. No code changes.**
> This document is the authoritative deep-dive for the seven expansion areas
> requested for v0.1.46. The seven VPCC workflow docs (PRD/PLAN/DESIGN/AGENTS/
> HANDOFF/CHANGELOG/VERSIONING) carry summaries and cross-reference this file.
> Nothing here is built yet; every item is a plan.

---

## Guiding principle (applies to all sections)

**Optimize for user experience and answer quality first. Token cost is not a
constraint.** Where a better answer is possible with a larger model, a
multi-model ensemble, a multi-pass refine-and-verify loop, or an extra
retrieval/tool round-trip, take it. The only budgets that matter are latency
(keep it acceptable) and correctness (never fabricate China-specific facts).

This reverses the v0.1.45 framing of "route factual messages away from the LLM
to save cost." We keep the intent classifier — but now its purpose is
**quality and correctness** (route to the right specialist model / the right
verified data source), not cost savings.

## 1. Multi-model Chinese LLM orchestration

### Why

A single general model (DeepSeek V4 Flash today) is a generalist. China travel
planning spans distinct sub-problems that different Chinese models are
individually stronger at: itinerary reasoning, Chinese-language POI/menu
understanding, very long context (multi-week trips + full chat history),
and China-specific factual knowledge (visa rules, transit, regulations).

### Candidate models (all Chinese providers, server-side keys only)

| Model | Provider | Strength / role in VisePanda |
|---|---|---|
| DeepSeek V4 / `deepseek-chat` | DeepSeek | Itinerary reasoning, tool-calling loop, JSON structure |
| Qwen (`qwen-max`, `qwen-mt`, `qwen-vl`) | Aliyun Bailian (key already configured) | Chinese POI/menu comprehension, translation, vision/OCR |
| GLM-4.x | Zhipu AI (智谱) | Long-context synthesis, alternative reasoner for cross-checks |
| Kimi (Moonshot) | Moonshot AI | Very long context (full trip + all messages), summarization |
| ERNIE (文心) | Baidu | China-specific knowledge (regulations, local facts) |
| MiniMax / Doubao | MiniMax / ByteDance | Optional additional voices for ensemble/judge |

### Orchestration patterns

1. **Intent-based routing** — the classifier picks the specialist:
   - `create_trip` / `adjust_trip` → DeepSeek (reasoner) with tool calling.
   - `ask_recommendation` about Chinese food/POI → Qwen (best at Chinese entities) after Amap/Dianping tool calls.
   - `ask_factual` about visa/regulations → ERNIE or a verified static source, cross-checked.
   - Long-history summarization / "what did we decide" → Kimi (long context).
2. **Parallel ensemble + judge** — for high-stakes answers (final itinerary, visa
   eligibility), query 2–3 models in parallel, then a judge model (or a
   deterministic reconciler) merges/selects the best answer and flags
   disagreements as `watchOut` items. Cost is irrelevant per the guiding
   principle; correctness wins.
3. **Refine-and-verify loop** — draft with the reasoner, then a second pass with a
   different model verifies facts (dates, prices, opening hours from tool data)
   and rewrites to the normalized `{headline, body, highlights, watchOut, nextStep}`
   schema.
4. **Graceful fallback chain** — every model has a fallback; if all live models
   fail, fall back to the existing mock Butler. Never remove the mock fallback.

### Architecture

- A `lib/ai/modelRegistry.ts` describing each provider (base URL, model id, env
  key name, capabilities: `reasoning | chinese | longContext | vision | judge`).
- A `lib/ai/orchestrator.ts` that, given an intent + context, selects the
  pattern (single / ensemble / refine-verify) and the model(s).
- Each provider wrapped behind a common `ChatCompletionProvider` interface so
  the orchestrator is provider-agnostic. All keys server-side:
  `DEEPSEEK_API_KEY`, `DASHSCOPE_API_KEY`, `ZHIPU_API_KEY`, `MOONSHOT_API_KEY`,
  `ERNIE_API_KEY` / `BAIDU_*`, `MINIMAX_API_KEY`. None exposed to the browser.
- OpenAI-compatible endpoints where available (DeepSeek, Zhipu, Moonshot,
  MiniMax) reuse one HTTP client; Qwen/DashScope keeps its existing helper;
  ERNIE uses its own auth (access-token) helper.

### Decision → ADR-044.

## 2. Native iOS App + Android APK (plan only)

### Goal

Real native apps, not a WebView shell around go2china.space. Native navigation,
native camera/mic, offline-first travel data, push notifications, app-store
distribution.

### Stack decision

Two viable paths:

| Path | Pros | Cons |
|---|---|---|
| **A. React Native + Expo** (recommended) | Reuses the existing React/TypeScript logic, types (`lib/types/trip.ts`), i18n, and provider abstractions; one team; true native UI via RN components; Expo handles builds/OTA | Not "100% native code"; heavy native modules need config plugins |
| B. Fully native (SwiftUI + Kotlin/Compose) | Maximum platform fidelity/performance | Two separate codebases; no logic reuse with the web app; slowest to ship |

**Recommendation:** React Native + Expo. It is genuinely native (compiles to
native views, not a web shell) while letting the existing TypeScript domain
layer, API routes, and i18n be shared. If the user specifically requires
Swift/Kotlin native, path B is documented but doubles the effort.

### Architecture

- **Backend reuse:** the existing Next.js API routes (`/api/chat`,
  `/api/explore/*`, `/api/translate/*`, `/api/exchange-rate`, future `/api/admin/*`)
  become the mobile backend unchanged. The app is a native client of the same API.
- **Shared core package:** extract `lib/types`, `lib/i18n`, provider interfaces,
  and orchestrator client into a shared workspace package consumed by both web
  and mobile.
- **Native UI:** RN screens mirroring Chat / Trips / Explore / Tools / Account;
  bottom tab navigator; native Chat composer; native map (react-native-maps +
  Amap SDK for China).
- **Offline-first:** cache trips, tools checklists, and the phrasebook in local
  SQLite (WatermelonDB / expo-sqlite) so travelers on the ground without signal
  keep access. Sync on reconnect.
- **Native device features:** camera (OCR menu scan), microphone (STT), push
  notifications (visa deadline, booking reminders, day-of alerts), share sheet,
  deep links (`visepanda://trip/:id`), biometric unlock.
- **Auth:** Supabase supports React Native; reuse the same auth + RLS.

### App-store considerations

- Apple: Apple Developer Program ($99/yr), App Store review, privacy nutrition
  labels, sign-in-with-Apple required if other social logins exist.
- Google: Play Console ($25 one-time), data-safety form, APK/AAB signing.
- **China distribution:** Chinese Android stores (Huawei, Xiaomi, OPPO, vivo,
  Tencent MyApp) require an ICP filing (备案) and often a software copyright
  (软件著作权). iOS in mainland China requires the app to be registered with
  MIIT. Plan for a China-specific release track separate from global Play/App
  Store. This is a legal/ops workstream, not just engineering.

### Phasing

1. Extract shared core package (web unaffected).
2. Expo app skeleton + auth + Chat (thin client over `/api/chat`).
3. Trips + offline cache.
4. Tools + Explore + native map.
5. Camera OCR + mic STT + push notifications.
6. Store submission (global first, then China track with 备案/软著).

### Decision → ADR-045.

## 3. Tools functional upgrade — real capability, not text

Today the six Tools are static checklists (Currency alone has a live rate).
Target: each becomes interactive and does something.

| Tool | From (text) | To (functional) |
|---|---|---|
| **Visa & entry** | Checklist copy | **Interactive eligibility checker**: pick nationality → get visa type / visa-free / 144h–240h transit eligibility; a transit-time calculator; a document checklist with saved progress state |
| **Payment setup** | Checklist copy | **Setup wizard**: step-by-step Alipay / WeChat Pay international-card linking with deep links to the apps; card-compatibility checker; "test a ¥1 payment" guidance |
| **Currency** | Live CNY rate line | **Full converter**: amount input, live rate (existing `/api/exchange-rate`), reverse convert, common-amount quick chips, offline last-known-rate snapshot |
| **Metro** | Checklist copy | **Route planner**: city + from/to station search → route, transfers, fare, time via Amap transit API (`/api/tools/transit`); station name in EN + 中文 for showing drivers |
| **eSIM / VPN** | Checklist copy | **Actionable options**: compare eSIM providers with real purchase links, data/price table, QR install guidance; VPN legality note + pre-arrival setup CTA |
| **Emergency** | Checklist copy | **One-tap actions**: call buttons for 110 / 120 / 119, GPS embassy locator, medical phrase cards with TTS, "share my location in Chinese" generator, nearest-hospital via Amap |

### Architecture

- Keep the `ToolsProvider` abstraction. Extend `ToolCategory` with an optional
  `interactive` descriptor (widget type + config) so cards can render an
  interactive component below the static content, degrading to text when data
  is unavailable.
- New server routes as needed: `/api/tools/transit` (Amap transit), reuse
  `/api/exchange-rate`. All third-party keys stay server-side.
- Interactive widgets are separate components (`components/tools/widgets/*`) so
  each tool can evolve independently.
- Emergency call/location and phrase TTS should also work offline where the OS
  allows (tel: links, cached phrase audio).

### Decision → ADR-046.

## 4. Account — professional UI + lead capture (留资)

### Professional account center

The current `AccountMenu` is a compact popover. Add a dedicated, mainstream,
professional account surface that signals trust and formality (this matters for
a product asking travelers for personal/travel details):

- A real `/account` page (in addition to the quick popover) with a clean,
  card-based profile center: avatar + name + membership tier, account security
  (email, password, connected Google), verified badges, and a profile-completeness
  meter.
- Sections: **Profile**, **My Trips summary**, **Preferences** (the
  `UserPreferenceProfile`), **Travel documents** (passport country, visa status —
  stored for the traveler's own convenience, encrypted, opt-in), **Notifications**,
  **Privacy & data** (export / delete, consent management).
- Visual tone: professional, formal, spacious; consistent with the Warm New
  Chinese system but more "banking/airline account" in structure and clarity.

### Lead capture (留资) — what to collect and how

Purpose: capture qualified traveler leads for follow-up / concierge upsell,
without a wall-of-forms that scares people off. Use **progressive profiling** —
collect a little at natural moments (after a trip draft, before saving, before
requesting a concierge quote), never all at once.

**Fields to collect (prioritized):**

Tier 1 — contact (minimum viable lead):
- Name
- Email (already have from auth)
- Phone number **and/or WeChat ID** (WeChat is often the real contact channel for China travel)
- Preferred contact channel + best time

Tier 2 — trip qualification (makes the lead actionable):
- Nationality / passport country (drives visa advice)
- Travel dates or rough month + trip length
- Party size + composition (solo / couple / family with kids / group)
- Cities / regions of interest
- Budget range (per person or total)
- Trip purpose (leisure / business / study / family visit / honeymoon)

Tier 3 — enrichment (optional, improves concierge value):
- Interests (from `UserPreferenceProfile`)
- Dietary restrictions, mobility needs
- Whether they need visa / flights / hotels booked for them
- How they heard about VisePanda (attribution)
- Marketing / follow-up consent (explicit opt-in, GDPR/PIPL aware)

**Storage:** a Supabase `leads` table (see §6), linked to the user when signed
in or captured anonymously with a session id for guests. Consent flags stored
explicitly with timestamp and source.

**UX rules:** every field optional except the one being asked for at that
moment; show why you're asking ("so we can check your visa route"); never block
core planning behind the form; a single "Talk to a China travel expert" CTA
opens the concierge lead form.

### Decision → ADR-047.

## 5. User management admin backend + LLM customer briefs

### Goal

An admin-only backend where staff can see captured leads and chat
conversations, with an LLM pass that distills each customer's raw data +
conversation into an efficient, structured brief so an admin understands the
customer in seconds instead of reading transcripts.

### Surface

- A role-gated `/admin` area (separate from the traveler app), accessible only
  to users with an `admin` role. Never shown in traveler navigation.
- **Leads dashboard:** sortable/filterable table of leads — name, contact,
  nationality, dates, budget, lead score, status (new / contacted / quoted /
  won / lost), source, created date. Bulk export (CSV).
- **Customer detail:** the full lead record + linked trips + full chat history,
  next to the **LLM-generated customer brief**.
- **Conversations view:** browse chat conversations, filter by intent/city/date.

### LLM customer-brief pipeline

For each customer, run their lead fields + chat messages + trip state through
the multi-model orchestrator to produce a structured brief:

```
CustomerBrief {
  summary            // 2-3 sentence who-they-are
  tripIntent         // cities, dates, length, party
  budgetSignal       // parsed budget + confidence
  readinessToBook    // score 0-100 + rationale
  keyPreferences     // food, pace, interests, dietary, mobility
  openQuestions      // what's missing to close (dates? visa? budget?)
  objections/risks   // hesitations detected in conversation
  suggestedNextAction// concrete recommended follow-up for the agent
  language           // preferred contact language
}
```

- Generated on demand and cached; regenerated when new messages/leads arrive.
- Uses a long-context model (Kimi/GLM) to read the whole conversation, then the
  reasoner to score readiness and propose next action. Cost is not a concern.
- The brief is advisory; raw data is always available beside it.

### Data & security

- New Supabase tables: `leads`, `lead_events`, `admin_users` (or a role column),
  `customer_briefs` (cached LLM output). RLS: travelers can only read/write
  their own lead; admins read across all via a policy checking the admin role.
- **Admin API routes run server-side only** and use `SUPABASE_SERVICE_ROLE_KEY`
  strictly on the server (never in browser code — standing constraint). Admin
  auth gated by Supabase session + role check on every request.
- PIPL/GDPR: personal data access is logged; consent and data-retention rules
  respected; export/delete honored.

### Decision → ADR-048.

## 6. Supabase schema additions (for §4–§5)

Planned new migration `0004_leads_and_admin.sql` (run in order in SQL Editor):

- `leads` — id, user_id (nullable), session_id, name, email, phone, wechat,
  preferred_channel, nationality, travel_month, trip_length_days, party_size,
  party_composition, cities jsonb, budget_range, trip_purpose, interests jsonb,
  needs jsonb, source, marketing_consent bool, consent_at, status, lead_score,
  created_at, updated_at.
- `lead_events` — id, lead_id, type (created/updated/contacted/quoted/won/lost),
  note, actor, created_at.
- `customer_briefs` — id, lead_id, brief jsonb, model_used, generated_at.
- `profiles` — the `UserPreferenceProfile` from the v0.1.45 roadmap (id, user_id,
  profile jsonb, updated_at).
- Role: add `role` to the user profile (`traveler` | `admin`) or a dedicated
  `admin_users` table; RLS policies for admin-wide read access.

All keys server-side; service-role key never in browser code.

## 7. Frontend & visual optimization

A dedicated design-quality track that runs alongside features.

- **Design tokens & system:** formalize the Warm New Chinese palette
  (`--paper`, `--ink`, `--cinnabar`, `--gold`, `--sage`) into a documented token
  set; consistent spacing/typography scale; a small reusable component library
  (Button, Card, Field, Pill, Modal, Sheet, Toast) so pages stop hand-rolling styles.
- **Motion & feedback:** purposeful micro-interactions (canvas patch animates in,
  day-card quick-actions, chat streaming), loading skeletons instead of blank
  states, optimistic UI on send/save, toast feedback.
- **Empty & error states:** every surface gets a designed empty state and a
  friendly error state (the first-run archetypes from the v0.1.45 roadmap fit here).
- **Accessibility:** color-contrast audit, focus-visible states, keyboard nav,
  reduced-motion support, RTL correctness (Arabic already supported).
- **Responsive polish:** continue the late-cascade mobile system; tablet
  breakpoint pass; ensure the new Account/Admin/Tools widgets are mobile-clean.
- **Performance:** image optimization (Next/Image, responsive art), route-level
  code splitting, font loading strategy, background-art weight budget.
- **Brand:** a small illustration/iconography system (panda mascot states,
  destination art) for a cohesive, premium feel.

### Decision → ADR-049.

## Consolidated build sequence (recommended)

These extend the v0.1.45 chat/data roadmap (阶段十二). Exact version numbers are
assigned when each is built; the **order** is the commitment.

1. **Multi-LLM orchestration** (§1) — foundational; upgrades answer quality for
   every downstream feature. Pairs with the v0.1.45 Chat Intelligence layer.
2. **Tools functional upgrade** (§3) — high, immediate traveler value; mostly
   self-contained.
3. **Account professional UI + lead capture** (§4) — unlocks the business model
   and feeds the admin backend.
4. **Admin backend + LLM briefs** (§5) — depends on lead capture + orchestration.
5. **Frontend & visual optimization** (§7) — continuous; front-load a design-system
   pass before the native app so mobile inherits a mature system.
6. **Native iOS/Android apps** (§2) — largest effort; sequence after the web
   feature set and design system stabilize so the shared core is worth extracting.

Parallelizable: §7 (design system) can run continuously; §1 can start immediately;
§2 legal/ops (备案/软著/developer accounts) can start in parallel because of lead times.
