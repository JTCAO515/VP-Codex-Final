# 用户分层判定逻辑 — 一次性游客 vs 重复/商务访客（设计笔记）

日期：2026-07-07
性质：轻量设计笔记，供未来实际排期"信任漏斗"商业化功能时参考，不是完整技术方案

## 背景

"信任漏斗"商业化方向（大部分免费，少量深度信任用户才转向定制化高客单价服务）经压力测试后，只对"重复/商务访客"这类可能一年多次往返的用户成立，对一次性游客（7-14天停留）在数学上不成立。要让这个判断可执行，需要一个低成本的信号，用来判断"该不该向这个用户展示定制化服务入口"。

## 判定信号（按获取成本从低到高排列）

1. **账号存在时长 vs 首次真实使用的间隔**：Supabase Auth 的 `created_at` 是现成的（`auth.users` 表标准字段，登录/注册相关代码已经在读取这张表），如果账号创建时间距今超过某个阈值（例如 3 个月）且近期又有活跃行为，是"回头客"的弱信号——不需要新增任何数据采集。
2. **行程创建次数**：`TripStore`（iOS）/对应 Android `TripRepository` 已经在每次创建行程时写 `createdAt`，统计一个账号名下有多少条独立行程记录，是比登录次数更强的信号（登录可能是误触，创建行程说明真的在规划一次新的旅行）。
3. **AI Profile 记忆条目的时间跨度**：现有的 AI Profile（`/butler/memory/profile`）已经在记录用户偏好，如果记忆条目的 `updatedAt` 时间跨度超过一次旅行的合理时长（例如超过 30 天），说明这不是单次旅行内产生的偏好，是跨行程积累的，可以作为重复访客的辅助信号，不需要新增采集，只是换一种方式读取已有数据。
4. **（需要新增，成本稍高）显式自选**：在 Arrival Setup 或 Me 页面让用户自己选一次"这是您第几次来中国"（首次/多次/常驻），比推断信号更准，但需要新增一个 UI 入口和一个数据字段，属于"轻量新增"而非"零成本推断"。

## 使用方式（只定义判定逻辑，不涉及具体转化 UI）

判定结果只用来回答一个问题：**"要不要向这个用户展示定制化服务入口"**，不涉及定制化服务本身长什么样（那需要操作者先想清楚履约方案，见 `final-product-positioning-moat-and-risk-assessment.md` 第 3.2 节的警告：履约方案不清楚之前，这个方向和此前被否定的"六个技术壁垒"是同一种自我欺骗）。

建议的最小判定规则（组合信号 1-3，不依赖信号 4）：
```
isLikelyRepeatVisitor =
  (账号存在时长 > 90 天) AND
  (行程创建次数 >= 2 次不同城市/不同日期范围的行程)
```
只有满足这个条件的用户，未来才会在 Me 页看到定制化服务的入口；不满足条件的用户，Me 页维持现状（不做任何"你还不够资格"这类用户能感知到的分级提示）。

## 隐私边界

- 判定过程完全在后端/本地推断，用户不应该看到任何"您的信任等级"这类展示。
- 不满足条件的用户不会看到任何被排除的提示——没有入口就是没有入口，不做"此功能仅限老用户"这种说明性文案，避免制造被区别对待的感觉。
- 这套信号全部来自已经在收集的数据（认证时间戳、行程记录、AI Profile 更新时间），不需要为了做这个判定新增任何额外的用户数据采集，符合项目一贯的最小化数据收集原则。

## 不在本笔记范围内的事

- 定制化服务具体是什么形态、谁来履约、怎么定价——这是操作者需要先想清楚的产品决策，不是这份笔记要回答的问题。
- 具体在哪个界面展示入口、展示成什么样的 UI——等履约方案确定后再设计。
