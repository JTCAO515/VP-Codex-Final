## Frontend Work Protocol

1. Read `PRODUCT.md`, `DESIGN.md`, `AGENTS.md`, and the relevant component/CSS files.
2. Classify the surface as product workspace, landing/brand, content/editorial, or admin. VisePanda product workspaces default to compact operational UI.
3. Reuse existing tokens, radii, icons, and layout primitives before adding CSS.
4. For new or revised controls, make the expected state complete: default, hover/focus, active, disabled, loading, empty/error, and mobile behavior.
5. Use lucide-react icons first. If a missing symbol is needed, search the optional icon resources and add the smallest consistent implementation.
6. Add motion only when it explains what happened. Keep durations short and support `prefers-reduced-motion`.
7. Before handoff, check no text overlaps, no mobile horizontal overflow, no body scroll on desktop one-viewport workspaces, and no hidden active controls behind the fixed mobile nav.
8. If multiple TSX components are edited, run the React best-practices review checklist before final validation.

## Explicit Non-Goals

- Do not overwrite the VisePanda visual system with a downloaded DESIGN.md.
- Do not add runtime npm dependencies only for design review.
- Do not introduce a new component framework unless a later implementation explicitly chooses one.
- Do not expose external design-tool or icon-tool credentials in the repo.
- Do not use external resources to justify fake product capability.
