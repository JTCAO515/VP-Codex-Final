# VisePanda — Product Interaction Blueprint (v0.1.52)

> Documentation-only strategic iteration. No product code changes.
> Purpose: rethink VisePanda from the full traveler journey, not from the current
> tab list. This document is the deeper product/interaction blueprint that
> should guide the next implementation rounds.

## 1. Product Thesis

VisePanda should not be positioned as "an AI itinerary generator." That is too
small and too easy to compare with generic ChatGPT prompts.

The stronger positioning:

> VisePanda is a one-stop AI travel butler for independent foreign travelers (FITs) in China, integrating itinerary planning, booking, translation, and local utility plugins (transit, payment, maps) into a live, actionable trip canvas.


This matters because the target user is not only asking "where should I go?"
They are trying to reduce five anxieties:

- Entry anxiety: visa, transit, passport, customs, documents.
- Payment anxiety: Alipay, WeChat Pay, cards, cash backup.
- Connectivity anxiety: eSIM, VPN, maps, translation availability.
- Language anxiety: menus, taxi addresses, signs, emergencies.
- Itinerary anxiety: route logic, fatigue, opening hours, real POIs, hotels.

The product should behave less like a set of pages and more like a butler with a
single memory:

- Chat is the command center.
- Trip Canvas is the source of truth.
- Tools are operational checklists and widgets.
- Translate is an everywhere utility.
- Explore is the discovery engine feeding Chat and Canvas.
- Trips are memory, status, sharing, and continuity.
- Account is trust, preferences, consent, and future lead capture.
- Community is proof and inspiration, not the main planning surface.

## 2. Core Interaction Loop

The product loop should be:

1. User expresses intent in natural language or chooses an archetype.
2. VisePanda extracts preferences and constraints silently.
3. VisePanda uses live tools/data where facts matter.
4. Canvas updates as the stable trip object.
5. Chat explains what changed and what decision remains.
6. User acts through small next-step controls, not by prompt engineering.
7. Trips saves the state and moves the user toward readiness.

The current app has most pieces, but the visible interaction still feels like
"six tabs." The next product work should make the loop visible and reduce tab
switching.

## 3. User Journey Model

### Stage A — "I am curious"

User mindset:

- "Is China realistic for me?"
- "What kind of trip should I even do?"
- "Will this be too hard?"

Best interaction:

- Landing page offers 3-5 high-confidence archetypes, not generic feature cards.
- User can start without typing: first China essentials, foodie China, family
  China, history and nature, business plus weekend.
- Each archetype opens Chat with a prefilled intent, not a finished static trip.

Success signal:

- User reaches a first useful canvas in under 60 seconds.

### Stage B — "I am planning"

User mindset:

- "Does this route make sense?"
- "Can you make it less tiring?"
- "Where should I stay?"
- "What is actually open and worth visiting?"

Best interaction:

- Chat remains the primary workspace.
- Canvas shows completeness and decisions still needed.
- Day cards expose quick actions: Lighten, Add food, Swap morning, Add hotel area.
- Explore results appear inline when the user asks for recommendations.

Success signal:

- User changes the plan 3-5 times without needing to understand prompts.

### Stage C — "I am preparing"

User mindset:

- "What must I do before I fly?"
- "Can I pay, connect, and communicate?"
- "What should I book now?"

Best interaction:

- Canvas/Trips exposes a preparation checklist derived from the trip.
- Tools appear as contextual cards: visa status, payment setup, eSIM, currency,
  metro, emergency card.
- Save to Trips should be paired with "Prepare this trip," not just persistence.

Success signal:

- User completes or acknowledges pre-departure blockers.

### Stage D — "I am in China"

User mindset:

- "Show this to a taxi."
- "Translate this menu."
- "What is near me?"
- "I am tired; change tonight."

Best interaction:

- Translate is a floating global utility, not a full tab the user must find.
- Day view becomes operational: Chinese address, map, phone, open hours, notes.
- Emergency and language tools work without breaking context.

Success signal:

- User can solve one real-world friction point in under 10 seconds.

### Stage E — "I want to share or get help"

User mindset:

- "Can someone review this?"
- "Can I share this plan?"
- "Can VisePanda or a human expert help me book?"

Best interaction:

- Trips status maps to user language: Taking shape, Ready for review, Travel-ready.
- Share link is easy but not dominant.
- Account/lead capture appears only at high-intent moments: after a strong draft,
  before concierge help, before document-sensitive checks.

Success signal:

- User saves, shares, or leaves contact details without feeling forced.

## 4. Page Role Redefinition

### Home

Current risk:

- Feature-card home tells users what the app contains, but does not help them
  start a trip.

Target role:

- Acquisition and first-start surface.
- Should answer: "Which China trip should I start with?"

Priority changes:

- Replace feature grid emphasis with archetype starts.
- If logged in and has active trip, route them toward Chat/Trips instead of
  showing marketing copy.
- Keep brand trust signals compact: AI China Travel Butler, live trip canvas,
  real China tools.

### Chat

Current role:

- Core workspace with model status and live canvas updates.

Target role:

- The operating center for planning and decision-making.

Priority changes:

- Add a first-run empty state with archetype chips.
- Render inline tool cards for factual needs: visa, payment, eSIM, metro.
- Promote `nextStep` from the structured Butler response into a primary chip.
- Show a small "remembered preferences" rail, but avoid making it feel like a
  form.
- Keep model/provider metadata subtle; travelers care about confidence, not
  model routing.

### Trip Canvas

Current role:

- Source-of-truth itinerary with daily cards and details.

Target role:

- The visible trip object, plus an action surface.

Priority changes:

- Replace generic "Live Trip Canvas" with trip title once available.
- Add trip completeness: route, hotel area, food, booking, tools, transport.
- Add quick actions on each day.
- Day detail should include operational data: Chinese name/address, open hours,
  phone, map, why this place fits the user's preference.

### Trips

Current role:

- Saved trip library and detail view.

Target role:

- Continuity, readiness, review, and sharing.

Priority changes:

- Status language should be user-facing: Taking shape / Ready for review /
  Travel-ready / Archived.
- Trip cards should show readiness blockers: missing hotel area, no payment
  setup, visa not checked.
- Detail page should keep itinerary primary and actions compact, as already
  repaired in v0.1.43.

### Explore

Current role:

- Standalone discovery page with city filters and Add to Trip.

Target role:

- Discovery engine that is mostly accessed from Chat and Canvas.

Priority changes:

- Keep the page for browsing, but stop treating it as equal to Chat.
- Add "In your trip" badges when POIs already appear in the active trip.
- Re-rank by preference profile.
- Make Add to Trip more precise: Add to Day, Replace morning, Use as dinner.

### Tools

Current role:

- Practical static checklist drawer cards, with currency partly live.

Target role:

- Operational widgets that resolve travel anxieties.

Priority changes:

- Tools should surface inside Chat when relevant instead of requiring navigation.
- First high-value widgets: visa eligibility, payment setup wizard, currency
  converter, metro route planner, emergency card.
- Each tool should support "mark as done" or "save to trip prep" later.

### Translate

Current role:

- Dedicated translator page.

Target role:

- Global travel utility.

Priority changes:

- Keep full page for deep use.
- Add floating camera/mic action in the app shell.
- From Day detail, provide "show Chinese address" and "translate nearby" entry.
- Voice/OCR should return to the previous page after use, not strand the user.

### Community

Current role:

- Local interactive MVP with feed, hotspots, photos.

Target role:

- Social proof and inspiration, not the planning spine.

Priority changes:

- Community posts should be convertible into a Chat planning prompt.
- Public shared trips become the best community content type.
- Hotspots should show whether they match the user's preference profile.

### Account

Current role:

- Lightweight auth popover.

Target role:

- Trust center, preference center, and lead capture.

Priority changes:

- Add a real `/account` center later, but keep popover.
- Preference profile should be visible and editable.
- Lead capture should appear progressively and explain why each field matters.
- Do not collect passport/visa-sensitive fields without explicit opt-in.

## 5. Feature Linkage Matrix

| User action | Surface | Behind-the-scenes linkage | Result |
|---|---|---|---|
| Starts with archetype | Home → Chat | Archetype prompt + preference seed | First canvas quickly |
| Asks "best local food in Nanjing" | Chat | Intent classifier → Amap/Dianping → Qwen/DeepSeek | Real POI suggestions |
| Taps Add to Trip in Explore | Explore → Chat | `/chat?add=` existing pipeline | AI rebalances route |
| Taps Lighten Day 2 | Canvas → Chat | Prebuilt adjustment intent | Canvas patch updates |
| Asks about visa | Chat | Tools knowledge card + optional alert | No tab switch needed |
| Saves trip | Chat/Trips | Supabase trip + canvas version | Continuity |
| Opens trip detail | Trips | Canvas summary + readiness blockers | Review state |
| Scans menu | Floating Translate | OCR/TTS route | Return to current context |
| Requests expert help | Account/Trips | Lead capture + consent | Admin follow-up |

## 6. Prioritized Roadmap

### v0.1.54 — Interaction Shell I: first-run and next-step UX (implemented)

- Home archetype starts.
- Chat empty state with 3 guided starts.
- `nextStep` from structured replies becomes a primary action chip.
- Canvas uses trip title and user-facing confidence labels.

Why first:

- It improves the first 60 seconds and makes the existing intelligence visible.

### v0.1.55 — Canvas Action Layer

- Trip completeness score.
- Day quick actions: Lighten, Add food, Swap morning, Add rest.
- Preparation blockers from Tools and trip state.
- Day detail operational fields planned but still optional.

Why second:

- It turns Canvas from display into a control surface without adding new external
  dependencies.

### v0.1.56 — Inline Tool Cards

- Visa, payment, eSIM, currency, and emergency cards render inside Chat.
- `ask_factual` can answer from Tools data without full itinerary mutation.
- "Add reminder" and "Mark as done" planning hooks.

Why third:

- It directly attacks traveler anxieties and reduces tab switching.

### v0.1.57 — TripBlock POI Embedding + Day Detail Upgrade

- Add optional POI fields to `TripBlock`.
- Day detail shows rating, hours, phone, address, photo, map/deeplink.
- Explore cards can show "In your trip."

Why fourth:

- It makes real POI data persistent in the itinerary instead of temporary prompt
  context.

### v0.1.58 — Translate Everywhere

- Floating Translate action in AppShell.
- OCR bottom sheet / voice sheet.
- Day detail "show Chinese address" and "speak this" affordances.

Why fifth:

- It supports in-China usage and makes Translate a utility instead of a page.

### v0.1.59 — Tools Widgets I

- Currency converter.
- Visa eligibility checker.
- Payment setup wizard.
- Emergency card generator.

Why sixth:

- These are high-confidence utilities with clear user value.

### v0.1.60 — Account Center + Preference Review

- Dedicated `/account`.
- Profile completeness.
- Preference editing and consent.
- Concierge lead CTA at high-intent moments only.

Why seventh:

- It supports business conversion after the planning experience feels valuable.

### v0.1.61 — Admin + Customer Brief Planning/Build

- Leads schema and admin role.
- Customer brief generation.
- Conversation/trip summary view.

Why eighth:

- Admin value depends on meaningful lead capture and chat history.

## 7. UX Writing Rules

- Replace developer status with traveler status.
  - Draft → Taking shape.
  - Refined → Looking good.
  - Ready to save → Travel-ready.
- Avoid "provider," "metadata," "API," and "confidence" in traveler-facing text.
- Every question should explain why it matters.
  - Bad: "What is your nationality?"
  - Better: "Which passport will you travel on? I need it to check visa and
    transit rules."
- Every AI reply should end with a concrete next action, not a vague invitation.
- Important warnings should be specific and tied to the trip.

## 8. Metrics To Watch

- Time to first canvas.
- Number of meaningful trip edits per session.
- Save-to-Trips rate after first canvas.
- Percentage of users who complete at least one prep item.
- Translate quick-action usage during mobile sessions.
- Share-link creation rate.
- Concierge lead CTA open rate and completion rate.
- Ratio of AI replies that include live POI/tool context.

## 9. Implementation Guardrails

- Chat remains the spine. New features should generally enter through Chat or
  Canvas before becoming standalone pages.
- Do not add direct canvas mutations outside the AI pipeline; quick actions send
  structured Butler intents.
- Keep all rich fields optional.
- Keep all keys server-side except explicitly public, domain-whitelisted display
  keys.
- Preserve fallback paths.
- Favor small contextual controls over large new panels.
- If a page does not answer a traveler question, remove or demote it.
