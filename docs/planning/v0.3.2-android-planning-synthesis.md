# v0.3.2 Android Planning Synthesis

## Scope

This release is planning-only. It audits the parallel Android APK planning uploaded to GitHub, distills the strongest ideas, corrects overreach or product conflicts, and merges that work with the Today-first native Android plan from the Codex local planning pass.

No Java, Kotlin, XML, Compose implementation, business logic, schema, API key, provider integration, or runtime behavior is changed in this version.

## Inputs Reviewed

1. `docs/planning/v0.3.1-android-native-spec.md`
2. `docs/planning/ux-design-and-layout-spec.md`
3. Current `PRD.md`, `PLAN.md`, `DESIGN.md`, `AGENTS.md`, `HANDOFF.md`, `CHANGELOG.md`, and `VERSIONING.md`
4. Existing web product state through `v0.2.17`: Chat/Canvas, Explore POI, Tools widgets, Translate, Supabase auth/trips, and provider fallbacks

## Executive Judgment

The parallel planning document is valuable because it pushes the project away from vague "mobile app someday" thinking and into concrete native Android concerns: Jetpack Compose, Room, DataStore, MVI, MapView lifecycle, permission scaffolding, offline-first repository boundaries, and device-level failure modes. That depth should be retained.

However, the plan also needs product-level consolidation before engineering starts. Its 4-tab `Canvas / Chat / Explore / Tools` model is technically clean but less aligned with the traveler's on-the-ground mental model than `Today / Butler / Plan / Explore / Tools`. Some proposed interactions, such as global background taxi-card triggers, complex foldable modes, and three-column expanded layouts, are too heavy for the first build. The next implementation must stay narrow enough to ship a credible APK foundation.

Final decision: **v0.3.2 adopts Kotlin + Jetpack Compose + Room/MVI as the Android-native planning direction, but organizes the product around a Today-first five-surface model.**

## What To Keep From The Parallel Agent

### 1. Kotlin / Jetpack Compose Direction

The parallel plan's strongest contribution is making Android native development concrete. The next implementation should treat Compose, Material 3, Navigation Compose, ViewModel, StateFlow, Room, DataStore, Hilt, Retrofit/OkHttp, WorkManager, CameraX, SpeechRecognizer, and Amap Android SDK as the evaluation baseline.

Reason: these choices map directly to the Android APK target and avoid building a mobile strategy around desktop web assumptions.

### 2. Offline-First Local Data Layer

Room + DataStore + repository boundaries should be retained. The app must work when a traveler is in a subway station, taxi, scenic area, hotel lobby, or airport queue with weak network.

Retained rule:

- Remote trip state is canonical after login.
- Local Room cache is the latest usable travel copy.
- DataStore owns lightweight settings and guest preference state.
- UI/session state must not pretend to be saved itinerary state.

### 3. MVI / StateFlow Mental Model

The parallel plan correctly identified that native Android should avoid scattered mutable local state. Trip edits, checklist toggles, and cached data changes should flow through explicit view-model actions into repository updates and then back to UI state.

This complements the existing web rule: itinerary content changes must stay auditable and should not bypass the Butler/canvas-patch contract when they alter trip meaning.

### 4. MapView Lifecycle And Gesture Risk

The MapView lifecycle warning is important. Amap MapView must be lifecycle-managed and should not be allowed to recompose excessively. Map gestures, bottom sheets, and scroll containers need explicit interaction ownership.

Retained acceptance requirement:

- Any implementation that embeds Amap MapView must define lifecycle ownership, camera-update throttling, and gesture-conflict handling before it is merged.

### 5. Permission Scaffolding

The parallel plan's staged permission model should become a product rule:

- Do not request all permissions at launch.
- Ask for camera when the user opens scan translation.
- Ask for mic when the user opens voice input.
- Ask for location when opening nearby Explore or map/navigation handoff.
- Ask for notifications only when the user enables trip reminders.

### 6. API Route Reuse Boundary

The stage 5 route audit is practical and should be retained. Existing Next.js API routes remain the mobile backend where possible:

- `/api/chat`
- `/api/explore`
- `/api/explore/amap`
- `/api/tools`
- `/api/translate/text`
- `/api/exchange-rate`
- `/api/trips`

OCR/STT file or audio upload compatibility needs a focused validation before native integration.

## What To Correct Or Defer

### 1. Four Tabs Are Too Technical

`Canvas / Chat / Explore / Tools` reflects implementation surfaces, but a traveler thinks in moments: what to do now, ask the butler, inspect the plan, discover nearby options, use survival tools.

Correction:

- Use `Today / Butler / Plan / Explore / Tools` as the product-level navigation model.
- The old `Canvas` concept becomes the content model inside Today and Plan.
- `Chat` becomes Butler.

### 2. Native Foundation Should Not Start With Full Map Complexity

Amap Android SDK matters, but putting MapView at the center of the first foundation build risks turning v0.3.3 into a map-integration project instead of an app foundation project.

Correction:

- v0.3.3 should create the app shell, navigation, state/data skeleton, local mock trip adapter, and static screen skeletons.
- Amap MapView can be spiked separately or included only as a placeholder contract unless the build remains small.

### 3. Global Background Taxi-Card Triggers Are Removed

Fast taxi-card access is real user value. But hidden global triggers introduce permission, OS-policy, battery, privacy, discoverability, and accidental-trigger concerns.

Correction:

- First implementation: Today quick action + Day Detail action + Plan context action.
- Do not use hidden global triggers or OS-level shortcut behavior as product requirements.
- If later research proves a faster launch path is necessary, evaluate a normal Android home-screen shortcut or contextual trip reminder separately, with explicit user opt-in.

### 4. Foldable And Three-Column Layouts Are Over-Scoped

Foldables are useful but not the primary first-release device class.

Correction:

- Standard Android phone is the primary target.
- Large screens can share layout constraints, but no three-column production requirement before the phone experience is stable.

### 5. Real-Time Native Translator Should Not Block The First APK Shell

CameraX/SpeechRecognizer collection is worth validating, but translation is a later high-value module after Today/Plan foundation.

Correction:

- Translator is a v0.3.6 priority.
- v0.3.3 and v0.3.4 may include static tool entry points and permission-state designs, not full translation implementation.

### 6. Booking And Payment Must Stay Non-Transactional

The product can show booking readiness, candidate links, fee cautions, and provider handoff ideas. It must not imply checkout, inventory, refund, or payment responsibility before a real transaction boundary exists.

Correction:

- Labels should use "booking readiness", "info only", "needs confirmation", and "open provider".
- Avoid "book now" unless a future version implements full transaction support.

## Merged Product Model

### Primary Surfaces

1. **Today**
   - Default post-onboarding screen.
   - Shows current trip, city/day, next action, Now/Next/Later, offline status, and urgent actions.
   - Fast actions: Translate, Ask Butler, Show Taxi Card, Open Map.

2. **Butler**
   - Conversational command center.
   - Uses structured response blocks, change digest, next-step actions, and context chips.
   - Trip-changing actions remain auditable.

3. **Plan**
   - Full itinerary surface.
   - Owns day list, Day Detail, candidate POIs, booking readiness, prep blockers, taxi cards, and local checklist state.

4. **Explore**
   - Curated discovery for attractions, food, hotels, and activities.
   - Adds candidates to Plan rather than silently mutating committed itinerary slots.

5. **Tools**
   - Survival utility hub.
   - Translation, payment setup, transport, currency, eSIM/connectivity, entry checklist, emergency, and offline pack.

### Secondary Surfaces

- **Home / Onboarding**: one-time first-start flow and archetype launcher, not a bottom tab.
- **Trips**: saved trips and history, accessed through Plan/profile until multi-trip switching becomes a high-frequency native behavior.
- **Account**: profile/settings/trust surface, accessed from top avatar/profile.
- **Community**: explicitly deferred.
- **Admin**: not part of the traveler APK.

## Merged Native Architecture Direction

### Android Stack Baseline

The next implementation should evaluate and likely use:

- Kotlin
- Jetpack Compose
- Material 3
- Navigation Compose
- ViewModel + StateFlow
- Room for trip/offline cache
- DataStore for lightweight settings
- Hilt for dependency injection
- Retrofit + OkHttp for existing API routes
- WorkManager for eventual background sync
- CameraX and SpeechRecognizer for later Translator capture paths
- Amap Android SDK for later map/native handoff validation

### State Layers

1. **Canonical remote state**
   - Supabase trips/messages after login.
   - Existing API routes and RLS remain the backend contract.

2. **Local durable state**
   - Room cached trip, day detail, taxi card data, offline phrasebook/tool notes, provider snapshots.

3. **Lightweight local settings**
   - DataStore language, session hints, guest preference profile, last active trip id.

4. **Temporary UI state**
   - Open sheet, selected tab, loading states, permission dialog state, current map marker.

### Sync Principle

Do not solve full conflict resolution in the first foundation build. Define the model and local mutation journal early, but connect live sync only after the Today/Plan MVP is stable.

## Revised Version Roadmap

### v0.3.3 — Android Native Foundation

Goal: create the native Android project foundation and prove the app can run as an Android APK shell.

Scope:

- Project/package identity and build configuration.
- Compose + Material 3 app shell.
- Five-surface navigation: Today / Butler / Plan / Explore / Tools.
- Static screen skeletons with empty/loading/offline/error placeholders.
- Local mock trip adapter.
- Room/DataStore/repository interfaces defined but only minimally wired.
- No real booking, payment, map, camera, voice, or Supabase sync requirement.

Acceptance:

- A tester can install/open the APK, switch all five surfaces, view a mock trip, open a Day Detail placeholder, and see static taxi-card/tool entry points.
- No API key is required.
- Missing network does not break the shell.

### v0.3.4 — Today + Plan Execution MVP

Goal: make the APK useful without live AI.

Scope:

- Today screen with current trip/day/next action.
- Now / Next / Later timeline.
- Plan day list and Day Detail.
- Taxi Driver Card with bilingual copy/display.
- Needs Scheduling candidate state.
- Booking readiness labels.
- Offline critical cards from local mock/cache.

Acceptance:

- A traveler can answer "what do I do now?", inspect the day, show a taxi card, and understand what is scheduled vs. candidate vs. needs confirmation.

### v0.3.5 — Butler + Sync Bridge

Goal: connect the native experience to the existing intelligence and persistence spine.

Scope:

- `/api/chat` integration.
- Existing `CanvasPatch` / structured assistant response contract mapped to native state.
- Supabase auth/trips/messages bridge.
- Guest local draft migration after login.
- Butler change digest and next-step action.
- Mock fallback remains available when network or keys fail.

Acceptance:

- The native app can ask the Butler to adjust a trip, show what changed, and save/sync after login without bypassing the existing auditable contract.

### v0.3.6 — Native Translator Utility

Goal: prove mobile-only value through text/camera/voice translation surfaces.

Scope:

- Text translation via existing route.
- Camera capture validation.
- Voice input validation.
- Phrasebook and full-screen show-card mode.
- Permission denied and offline fallback states.

### v0.3.7 — Explore + Candidate Pipeline

Goal: make discovery feed structured Plan candidates.

Scope:

- POI cards for attractions/food/hotels/activities.
- Existing Explore/Amap API route consumption.
- Add to Plan, Save for Later, Needs Scheduling.
- Source/confidence/fit rationale.
- No checkout.

## Non-Negotiable Rules For The Next Coding Agent

1. Build a native Android APK direction first; do not continue web feature expansion as the primary roadmap.
2. Do not build a webpage wrapper or hybrid shell.
3. Keep existing Next.js API routes as backend contracts where possible.
4. Keep all keys server-side; never commit API keys.
5. Preserve mock/static fallback behavior.
6. Do not imply booking/payment transaction capability before a real transaction boundary exists.
7. Do not let Explore or local UI directly mutate itinerary meaning; meaningful trip changes must remain auditable.
8. Treat Today and Plan as the first user-value proof before advanced map, translator, or booking surfaces.

## Final Summary

The other agent's plan contributes native Android engineering rigor. The Codex plan contributes traveler-centered product hierarchy and version sequencing. The merged direction is stronger than either alone:

- **Native stack clarity:** Kotlin + Compose + Room/MVI as baseline.
- **Product clarity:** Today-first five-surface navigation.
- **Execution clarity:** Foundation -> Today/Plan -> Butler/Sync -> Translator -> Explore.
- **Risk clarity:** map lifecycle, permissions, offline state, provider fallback, booking trust boundaries.

This is the new source-of-truth planning bridge from `v0.3.1` strategy into `v0.3.3` implementation.
