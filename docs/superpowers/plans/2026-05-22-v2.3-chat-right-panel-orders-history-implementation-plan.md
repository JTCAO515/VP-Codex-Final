# v2.3 /chat 右侧面板（历史 + 订单）Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在 `/chat` 页面新增右侧详情面板，支持登录用户查看行程版本、酒店订单、RFP/报价/服务订单历史；并对后端订单类接口做登录强制与用户隔离（基于 Supabase JWT）。

**Architecture:** 前端保持静态（Vanilla JS），在 `/chat` 增加一个“Details”面板，通过新增的 `/api/trips/{trip_id}/*` 接口加载数据；后端在现有模型上采用“沿链路回溯归属”（Trip.user_id / RFP.user_id）做隔离，不强行给每张表加 user_id 字段；所有订单类接口必须登录（Bearer token）。

**Tech Stack:** FastAPI, SQLAlchemy, Supabase Auth (Google OAuth), Supabase Postgres, Vercel rewrites, Vanilla HTML/CSS/JS.

- [ ] **Step 2: 为 orders 类接口接入 `get_principal(request, guest_id=None)`**

约定：
- 所有订单类接口（rfps、service-orders、hotel bookings）都必须登录
- 失败返回 401

示例（在各 router 顶部引入）：
```python
from fastapi import Request
from app.auth import get_principal

def _require_user(request: Request) -> str:
    p = get_principal(request, guest_id=None)
    return p.user_id
```

- [ ] **Step 3: 改造 `POST /rfps`：忽略 body.user_id，使用 token.sub**

把 `RfpCreateIn` 改为：
```python
class RfpCreateIn(BaseModel):
    trip_id: str
    service_types: list[str] = Field(default_factory=list)
    requirements: dict = Field(default_factory=dict)
```

在 handler 中：
- `user_id = _require_user(request)`
- 校验 trip 存在且 `trip.user_id == user_id`（否则 404）
- 创建 rfp：`rfp.user_id = user_id`

- [ ] **Step 4: 改造 `GET /rfps/{rfp_id}`：只允许 rfp.user_id == current_user_id**

- [ ] **Step 7: 补齐 tests（跨用户访问）**

由于单测里无法方便生成真实 supabase token，建议在测试环境（PYTEST）允许一个“测试 token”机制：
- 若 `AUTH_TEST_BYPASS=1` 且 header `Authorization: Bearer test:<user_id>` → 直接视为登录 user

在 `backend/app/auth.py` 增加：
```python
if token.startswith("test:") and os.getenv("AUTH_TEST_BYPASS") == "1":
    return Principal(mode="user", user_id=token.split(":",1)[1])
```

然后补充跨用户测试：
```python
def test_rfp_isolated_by_user():
    with TestClient(create_app()) as c:
        import os
        os.environ["AUTH_TEST_BYPASS"] = "1"
        # user A create trip via chat (guest->legacy not allowed), so directly create trip in DB is hard.
        # Instead: call /chat/messages with legacy user_id for test only.
```

> 注：如果你不想增加 bypass，也可以把订单类测试只覆盖“未登录 401 + 逻辑路径不 crash”，跨用户隔离留到 staging 手测。但为了安全，本计划推荐 bypass。
