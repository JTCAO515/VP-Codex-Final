# VisePanda 交接文档重构设计

## 概述

当前 `HANDOFF.md` 已经比普通项目说明更完整，但用户明确希望“内容更加详尽”，同时又不希望主交接文档无限膨胀、变成难以阅读的长文档。

因此，本轮不采用继续单纯扩写 `HANDOFF.md` 的方式，而采用：

> **主文档 + 附录索引**

也就是：

1. `HANDOFF.md` 继续作为主交接入口
2. 主文档负责建立全局认知和接手顺序
3. 更细的工程细节、接手建议和补充说明独立为附录
4. 主文档通过清晰索引把读者导向对应文档

## 目标

本轮完成后，文档体系需要达到以下效果：

1. 新接手的人在 10 分钟内能理解项目是什么、当前做到什么、当前问题是什么
2. 新接手的开发者能快速找到应该先看的文件、先跑的命令、先读的文档
3. 主交接文档不再承担所有细节，而是承担“导航与判断”的角色
4. 文档体系适合后续继续维护，而不是每次都重写整份 `HANDOFF.md`

## 非目标

本轮不做以下事情：

- 不修改产品功能
- 不修改测试
- 不改部署
- 不补新的工程实现
- 不把 README/CHANGELOG 全部重写

本轮聚焦在文档结构和交接体验本身。

## 方案选择

### 方案 A：继续扩写现有 `HANDOFF.md`

优点：

- 最省改动
- 保持现有连续性

缺点：

- 会让主文档越来越长
- 产品、工程、部署、问题会混在一起
- 新接手的人反而更难快速定位

### 方案 B：双层合并版

把产品层和工程层一起塞进 `HANDOFF.md`，通过分区来解决。

优点：

- 信息最全
- 不需要新增附录文档

缺点：

- 主文档仍然会很长
- 越往后越难维护

### 方案 C：主文档 + 附录索引

这轮采用的方案。

优点：

- 主文档保持可读性
- 附录承载更详尽的工程内容
- 最适合正式交接和后续维护

缺点：

- 需要新增至少一个附录文档
- 文档之间需要做好索引，不能只靠读者自己找

## 设计方案

## 主文档：`HANDOFF.md`

`HANDOFF.md` 继续保留为项目交接总入口，但职责会收敛为：

### 1. 项目是什么

- 项目定位
- 目标用户
- 当前产品性质
- 与通用 AI 网站的区别

### 2. 当前能做什么

- 页面结构
- 用户可见能力
- 工程能力
- 已达成效果

### 3. 当前还没做完什么

- 当前真实问题
- 当前风险
- 当前边界

### 4. 过往关键迭代

- 版本阶段性总结
- 最近几个关键版本做了什么

### 5. 当前部署与商用判断

- 当前保留 `Vercel`
- 当前不整站迁移
- 商用升级路线已有文档

### 6. 接手建议

- 新接手的人第一天先做什么
- 先看哪些文件
- 先跑哪些命令
- 不要先动哪些部分

### 7. 附录索引

- README
- CHANGELOG
- 商用升级方案
- 稳定性专项 spec / plan
- 工程接手附录

主文档的定位是：

> **帮助接手者快速建立全局认知、判断优先级、找到下一步入口。**

## 附录文档：工程接手附录

新增一个更偏工程接手的附录文档，建议命名为：

`docs/2026-06-20-engineering-handoff-notes.md`

该文档主要承担“详尽版工程交接”的职责，内容包括：

### 1. 当前主链路文件

- 哪些文件是当前活跃入口
- 哪些文件是兼容层
- 哪些文件看起来存在但不是主链路

### 2. 前端结构说明

- `web/index.html`
- `web/app.js`
- `web/app.css`
- 各 view 的组织方式
- 主要 overlay / modal / nav / state hook

### 3. 后端结构说明

- `api/index.py` 的路由入口角色
- `api/auth.py` / `api/chat.py` / `api/cities.py` / `api/tools.py` / `api/visa.py`
- 哪些路由最关键
- 哪些地方最值得小心修改

### 4. 数据与资源说明

- `data/*.json` 的作用
- `static/img` 的用途
- 图片和工具数据的真实使用方式

### 5. 测试说明

- Python 测试入口
- Node 前端测试入口
- 当前测试覆盖的是哪类问题
- 当前还没覆盖什么

### 6. 已知工程风险

- `web/app.js` 过重
- Vercel 冷启动
- SSE / 图片 / auth 相关的敏感点
- 兼容层与主链路并存

### 7. 接手第一周建议

- 第 1 天看什么
- 第 2 天跑什么
- 第 3 天之后先做哪类问题

该附录的定位是：

> **让一个新开发者能更放心地继续写这个项目，而不是只拿到一份产品描述。**

## 预期改动文件

本轮预计只修改 / 新增以下文档：

- `HANDOFF.md`
- `docs/2026-06-20-engineering-handoff-notes.md`

可选更新：

- `README.md`

是否修改 README 只在以下条件下进行：

- 如果需要补一个“交接文档索引”入口
- 否则不强行改动 README

## 错误控制与边界

为了避免把交接文档写成重复内容堆叠，本轮遵循以下约束：

1. `HANDOFF.md` 不再重复抄长段工程细节
2. 已经存在的文档不重复复制全文，只做索引
3. 附录文档的目标是补“接手所需的新信息”，不是重复 README
4. 主文档必须能独立阅读，但细节应链接到附录

## 验收标准

本轮文档重构通过的标准是：

1. `HANDOFF.md` 读起来比当前版本更容易导航
2. 主文档能清楚回答“项目是什么、做到什么、还有什么问题、先看什么”
3. 工程细节被转移到独立附录中
4. 主文档中存在清晰的附录索引
5. 新接手者不需要在仓库里盲找入口

## 版本与交付方式

这轮不更新产品版本号，不新增产品能力。  
只会：

- 重构交接文档
- 新增工程接手附录
- 提交并推送到 GitHub
